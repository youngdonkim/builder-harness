이 단계는 **사용자가 Claude Design으로 고품질 프로토타입을 만들어 오게 *안내*하고, 받은 standalone.html을 프로젝트 루트에 저장해 *사용자 테스트로 검증*하는 단계**다. 빌드·디자인 수정은 **Claude Design**(외부 도구)이 SoT — 디자인을 픽셀·컴포넌트 단위로 이해·수정하는 건 Claude Design이 가장 잘한다. Claude Code(여기)는 *안내·수령·테스트 준비·자산 예외*만 한다.

**핵심 분업** — Claude Design = *주방*(요리를 다 함), Claude Code(여기) = *냉장고+검수대*(받아 보관·시연, 자산 예외만 손봄):
- **Claude Design** = 디자인·화면·동작을 *완성*해 **standalone.html** 하나로 내보냄. 큰 수정도 여기서 (재호출 → 새 standalone.html로 덮어쓰기).
- **Claude Code(여기)** = standalone.html 받아 프로젝트 루트에 저장 → 사용자 테스트 준비 + *자산·예외 미세 보정*(거의 없음) + 이 파일을 *MvpBuild(6단계)로 넘김*.

**디자인 품질은 *사후 점검*이 아니라 *설계 입력*으로 확보한다.** §2.1의 **두 전문가 렌즈**(Accenture Song 경험 전략 · Apple Senior UX Researcher 사용성)를 Claude Design 호출 *프롬프트에 처음부터* 넣어 그 렌즈로 디자인하게 한다 — 다 만든 뒤 까는 것보다 산출 품질이 높다.

## 1. 단계 목표

- **완성 프로토타입 = 프로젝트 루트의 `[projectName]-proto.html` 하나** (projectName은 프로젝트 폴더명). Claude Design의 **standalone.html** 출력 그대로 — 폰트·이미지·컴포넌트 소스(JSX/CSS/디자인 토큰)까지 전부 파일 안에 내장된 자급자족 단일 파일이라, 브라우저로 더블클릭하면 그대로 도는 *작동 데모이자 시각 정답지*다. 별도 zip 압축 해제·README·컴포넌트 폴더 없음 — **이 파일 하나가 전부**.
- **버전 없음, 항상 최신 하나만**: 수정될 때마다 같은 파일명으로 **덮어쓴다**. `prototype-ver-{N}/` 같은 버전 폴더를 만들지 않는다 — 지금 프로젝트 루트에 있는 그 파일이 곧 현재 SoT.
- **첫 버전**: 사용자가 외부 Claude Design에서 §2.1 input 3개(screen-design + market-research 요약 + 두 렌즈 블록)로 호출 → **standalone.html 다운로드** → Claude Code(여기)로 가져와 프로젝트 루트에 저장.
- **이후 수정**: 큰 방향을 바꿀 때는 Claude Design 재호출(새 standalone.html로 덮어쓰기). 자잘한 예외 보정만 현재 파일에서 in-place (거의 없음).
- **프로토타입은 사용자 테스트용** — 출시 코드가 아니다. 본격 빌드는 다음 MvpBuild(6단계)에서 *이 파일을 입력으로* 진행.

**SoT 책임**: SKILL.md §2.1 SoT 매트릭스에 따라 prototype이 SoT인 영역:
- 시각 디자인·layout·컴포넌트·완성 프로토타입 화면 + 디자인 토큰·룰 — `[projectName]-proto.html` 하나 (진입점이자 정답지이자 소스 전부)
- **이 파일이 생기는 순간부터 상태(빈·로딩·에러)·전환 명세까지 화면 영역 전체의 SoT가 이 파일로 승격**된다 (SKILL.md §2.3 "화면 SoT의 코드 승격"). `screen-design.md`는 이 파일의 *입력*이었을 뿐 — 이후 화면이 바뀌면 `screen-design.md`를 고치는 게 아니라 §2.2대로 이 파일(재호출로 덮어쓰기 또는 in-place)로 반영하고, 아이디어·가설이 바뀐 부분만 `proto-retro.md` 진입 스냅샷에 새로 적는다.

## 2. 진행 절차

### 2.1 첫 버전 — Claude Design 호출 input 준비 + standalone.html 받아 저장

> **이 단계 진입 전**: 사용자가 **외부** [Claude Design](https://support.claude.com/en/articles/14604416-get-started-with-claude-design)에서 호출·디자인 생성·standalone.html 다운로드까지 처리해 옴. Claude Code(여기)는 *호출에 관여하지 않는다*. 이 단계는 **받아 온 뒤**부터 시작하지만, 호출 *전*에 input 3개를 준비해 안내한다.

**외부 Claude Design에 들고 갈 input — 3개** (사용자가 진입 *전*에 준비):

1. `planning/mvp/screen-design.md` — **단일 SoT**. 전 화면 명세(진입·내용·행동·전환·상태·데이터)·플로우·핵심 시나리오 + *검증화면에서 다듬은 최종 카피까지 통합된 상태* (3-screen-design.md §3.1.1 sync 절차). Claude Design 메인 입력 — 명세한 화면을 그대로 디자인, 카피는 그대로 사용.
2. **최신 진입 스냅샷** — 최종 아이디어·비목표·검증 가설·북극성 (무엇·왜·누구·스코프·톤). `screen-design.md` 머리 스냅샷이 원문이거나, `변경 없음`이면 `planning/mvp/market-research.md` §최종 서비스 기획 요약 내용 (SKILL.md §2.3).
3. **두 전문가 렌즈 블록** (아래 §2.1.1 — 복붙). *처음부터* 이 렌즈로 디자인하라는 설계 원칙.

디자인 시스템·토큰은 Claude Design이 위 input에서 *스스로 생성*해 파일에 담는다 — 별도 brand 입력 없음.

**결과물은 반드시 standalone.html로 요청한다** — Claude Design 대화에서 "완성되면 standalone.html(자급자족 단일 파일, 외부 파일 참조 없이 폰트·이미지·컴포넌트 소스 전부 내장)로 내보내줘"라고 명시한다. 여러 파일로 쪼개진 산출물(별도 README·컴포넌트 폴더 등)은 받지 않는다 — 이 하네스는 그 형태를 쓰지 않는다.

> **🔁 검증화면(`wireframes/`)은 input으로 들고 가지 않는다**. 검증화면은 *3단계(ScreenDesign) 안 작업 비계*. 핸드오프 직전에 검증화면 카피가 `screen-design.md`로 일괄 sync되어 `screen-design.md` 자체가 *시각 단서 0 + 최신 카피 보존*된 단일 텍스트 SoT가 된다 (3-screen-design.md §2.5 와이어프레임 가드와 같은 메커니즘).

#### 2.1.1 두 전문가 렌즈 — Claude Design 프롬프트에 복붙

> *사후 점검이 아니라 설계 원칙으로 넣는다.* Claude Design 프롬프트 끝에 "아래 두 전문가 렌즈로 화면을 설계하라"고 붙이고 두 표를 그대로 전달. 프로젝트 무관 baseline(Layer 1)이라 다른 프로젝트에도 그대로 빌려 쓴다 — "예" 칸만 그 프로젝트 것으로 바꾼다.

**렌즈 1 — Accenture Song (경험·전략)**
> 최적화 대상: 거래가 아니라 *사람의 순간과 감정*. "기능 많은 화면"이 아니라 "한 사람이 가장 불안한 순간을 통과하게".

| 원칙 | 디자인 지시 | 예 (집내비) |
| --- | --- | --- |
| 페르소나 한 명 | 인구통계("20~30대") 말고 *가장 불안한 한 명의 순간*을 위해 설계 | 첫 자취 '지오' — 역량 0·불안 100 |
| 메타포 일관 | 제품이 내비면 *다음 한 걸음*만. 전체 지도·단계 나열 금지 | 12단계 나열 X, "지금 할 1개"만 크게 |
| Hero moment <60초 | 첫 세션에 *손에 잡히는 가치* 하나 즉시 | "30초 호구 체크" or "오늘 할 일 1개" |
| 복리 hook | 첫 세션에 "어, 이거 몰랐네" 하나 심기 → 재방문 씨앗 | 월세 세액공제·확정일자 같은 잠재가치 |
| 톤 | 친구·주치의. 진단서·전문용어 금지(불가피하면 즉시 풀이) | "혼자 헤매지 마" |

**렌즈 2 — Apple Senior UX Researcher (사용성·인터랙션)**
> 최적화 대상: *생각하게 만들지 마라(self-evident)*. 명료함 > 기능. 설명서 없이 첫 사용자가 통과.

| 원칙 | 디자인 지시 | 예 (집내비) |
| --- | --- | --- |
| 화면당 주행동 1개 | 가장 중요한 게 가장 크고 먼저. 나머지는 뒤로 | "다음 할 일" 1개 + 전체경로는 접기 |
| 점진적 노출 | 한 번에 다 보여주지 말고 *필요한 순간에* (렌즈1 메타포와 상호보강) | 단계는 도달했을 때 펼쳐짐 |
| 모든 상태 설계 | 빈·로딩·에러·성공 화면까지 — 빈 화면 방치 금지 | "후보 없어"도 안내 카드로 |
| 즉각 피드백 | 모든 탭·입력에 눈에 보이는 반응 | 저장됨 ✓·링크 저장됨 |
| Deference | 장식·chrome 최소, 정보가 주인공 | — |
| 물리 제약 | 엄지 도달 영역 + 터치타깃 ≥44pt | 하단 주행동 버튼 |
| 접근성 | 명도대비 확보 + *색에만 의존하는 신호 금지*(색맹) + 큰 글자 대응 | 위험을 빨강"만" 표시 X — 아이콘·텍스트 병행 |

**Claude Code(여기)에서 하는 일**:
1. 사용자가 받아 온 **standalone.html**을 **프로젝트 루트**에 `[projectName]-proto.html`로 저장 (§2.3 — SoT이자 테스트·단계 판별용 신호 파일 겸용, 별도 폴더·압축 해제 없음).
2. 사용자가 브라우저로 그 파일을 **더블클릭으로 열어** 시각·플로우 **직접 클릭 검증** (서버·빌드 불필요).
3. 피드백 받아 — 큰 변경이면 §2.2-A(Claude Design 재호출), 자잘한 예외면 §2.2-B(in-place 보정).

**프로토타입 정밀도 가드** — Claude Design 산출물을 *프로토타입 수준*으로 둔다 (최종 시각·코드 lock은 MVP에서):
- screen-design 화면 명세의 데이터 필드 + 최종 서비스 기획 요약 내용 첫 베타 케이스로 placeholder 데이터 고정
- 다크모드·다국어·반응형 디테일은 프로토타입 범위 밖
- 목적은 *플로우·구성 + 첫인상 검증*이지 *최종 시각·코드 결정*이 아님

### 2.2 수정이 필요할 때 — 대부분 Claude Design, 예외만 여기서

Claude Design이 디자인을 SoT로 완성하므로 **여기서 다듬을 일은 거의 없다.** 수정이 필요하면:

| 변경 종류 | 처리 |
|---|---|
| **A. 구조·디자인·동작 변경** (layout·컴포넌트·플로우·큰 수정·렌즈 미반영분) | **Claude Design 재호출** → 같은 대화에서 "최신 standalone.html 다시 만들어줘" → 받아서 `[projectName]-proto.html` **덮어쓰기**. *기본 경로.* |
| **B. 예외 미세 보정** (Claude Design이 못 하는 자잘한 것 — 특정 이미지 교체·자산 삽입) | 현재 `[projectName]-proto.html`에서 **in-place 편집**. 거의 없음 (§2.2.1 참조). |

**받은 뒤가 끝이 아니다 — 테스트 → 다음 단계 제안까지 이어간다.** A·B 어느 경로든 파일을 받아 `[projectName]-proto.html`을 덮어썼으면, 거기서 멈추지 말고: (1) 사용자에게 **브라우저로 열어 직접 클릭 테스트**해보라고 안내 → (2) 테스트 결과가 괜찮고 §3 완료 체크리스트가 충족되면 **AI가 먼저** "이제 5단계(ProtoRetro — 데모 후 인터뷰)로 넘어가자"고 제안한다. 사용자가 알아서 다음 단계를 요청하길 기다리지 않는다 (SKILL.md §1.3 "AI가 먼저 제안" 원칙).

**SoT 판정**: 프로젝트 루트의 `[projectName]-proto.html`이 유일하고 현재의 SoT — 버전 이력을 따로 안 남긴다 (필요하면 git 커밋 이력으로 충분).

#### 2.2.1 예외 보정 시 — 파일 직접 편집 (최소만)

`[projectName]-proto.html`(= Claude Design standalone)은 자체 번들 포맷이다 — 폰트·이미지·상당수 컴포넌트 로직이 **base64로 인코딩**돼 있어 예전 방식(React+Babel 원문을 그대로 검색·수정)보다 직접 편집이 어렵다. 실전 요령:

- **편집 가능 영역**: 파일 안 일부 텍스트(화면 카피 등)는 평문으로 남아있어 검색·치환이 가능하지만, 컴포넌트 로직·이미지·폰트는 base64라 손으로 고치기 어렵다.
- **원칙**: 정말 사소한 것(오타 수준의 카피 수정)만 여기서 직접 손대고, **로직·레이아웃·자산 교체는 전부 §2.2-A(Claude Design 재호출)로 보낸다.** 이전처럼 base64 리소스를 손으로 까고 다시 인코딩하는 작업은 하지 않는다.
- **검증**: **사용자가 직접 브라우저로 연다** (`open <file>`). Claude는 자동 도구로 *검증·사인오프* 하지 않는다 (CLAUDE.md 검증 룰). 단 디자인 *리뷰*가 필요하면 렌즈는 §2.1.1대로 Claude Design 호출에 *처음부터* 넣는 게 정석 — 사후 리뷰 루프를 여기서 돌리지 않는다.

**왜 예외만 손보나** — 프로토타입은 *사용자 테스트용*이고 Claude Design이 디자인을 SoT로 완성한다. 여기서 코드를 크게 고치면 (1) 사용자 테스트 데이터가 *복제 품질*에 오염되고 (2) 어차피 MVP에서 새로 빌드하니 *버려지는 중간물*이 된다. 큰 변경은 Claude Design 재호출이 깔끔하다.

### 2.3 프로젝트 루트 신호 파일이자 SoT — `[projectName]-proto.html`

- **파일명 규칙**: `[projectName]-proto.html` — 여기서 **`projectName`은 프로젝트 폴더명**이다 (이 저장소 최상위 디렉토리 이름 그대로. 예: 저장소 폴더가 `zipnavi`면 `zipnavi-proto.html`).
- **받아오는 법**: Claude Design에서 standalone.html을 다운로드할 때 이 이름으로 바로 저장하거나, 기본 파일명으로 받은 뒤 프로젝트 루트로 옮기면서 이 이름으로 바꾼다.
- **갱신 시점**: 첫 수령 시 + 이후 §2.2 A(재호출)·B(in-place)로 갱신되든 **매번 이 파일을 같은 파일명으로 덮어쓴다** — 버전 번호를 붙이지 않아 항상 "지금 테스트해야 할 최신본" 하나만 프로젝트 루트에 존재한다.
- **SoT다** — 이전 하네스 버전과 달리 이 파일과 별도로 보관하는 zip·README·컴포넌트 폴더가 없다. 이 파일 하나가 화면·인터랙션·토큰·컴포넌트 소스 전부를 담은 유일한 산출물이다.
- **왜 단일 파일인가**: standalone.html은 이미 폰트·이미지·컴포넌트 소스(JSX/CSS/토큰)를 전부 base64 등으로 내장한 자급자족 파일이라, 이걸 압축 해제해 폴더로 쪼개 봐야 같은 정보를 다른 형태로 중복 보관하는 것뿐이다. 폴더 구조 관리 비용 없이 파일 하나만 신경 쓰면 된다 — 사람은 더블클릭해서 열면 되고, AI는 `/idea-to-mvp` 재호출 시 이 파일 존재 여부로 지금이 막 프로토타입이 나와 테스트 라운드에 들어갈 시점인지 빠르게 판단한다 (SKILL.md §1.2 현재 단계 판별에서 함께 확인).

## 3. 완료 체크리스트

다음이 모두 충족되어야 다음 단계(Retro)로 진입.

- [ ] **프로젝트 루트에 `[projectName]-proto.html`이 있고, 브라우저로 더블클릭하면 정상 실행됨** (폰트·아이콘·지도 등 CDN 자원은 네트워크로 로드되는 게 정상 — 파일 자체는 별도 로컬 폴더 의존 없이 자급자족).
- [ ] **screen-design 전 화면 명세가 화면으로 구현됨** — 명세한 화면 누락 0, 각 화면의 상태·전환이 시각화됨. 시나리오↔화면 추적 가능.
- [ ] **두 렌즈가 반영됨** — Claude Design 호출에 §2.1.1 렌즈를 넣었고, 결과 화면이 렌즈 원칙(주행동 1개·점진 노출·hero moment·접근성 등)을 위배하지 않음. 위배 시 §2.2-A 재호출.
- [ ] (사용자 검증) 사람이 브라우저로 `[projectName]-proto.html`을 클릭하며 검토 완료.

체크리스트 미충족 시 SKILL.md §4 단계 진입 평가 정책에 따라 사용자에게 선택지 제시.

## 4. 산출물 스펙

위치: **프로젝트 루트**의 `[projectName]-proto.html` 파일 하나. `planning/mvp/`엔 이 단계 산출물이 없다.

```
(프로젝트 루트)
├── CLAUDE.md
├── [projectName]-proto.html       ← 이 단계의 유일한 산출물이자 SoT (projectName = 프로젝트 폴더명, 항상 최신본 하나만)

    ├── market-research.md
    ├── screen-design.md
    ├── wireframes/                    ← 3단계(ScreenDesign) 검증 화면 (와이어프레임 비계)
    └── proto-retro.md
```

**파일 명명 원칙**:
- **`[projectName]-proto.html`은 SoT이자 프로젝트 루트의 신호 파일** — 상세는 §2.3.
- 버전 폴더·zip·README·컴포넌트 폴더 없음. 파일이 바뀌면 같은 이름으로 덮어쓴다.

## 5. MvpBuild(6단계)로의 핸드오프

이 `[projectName]-proto.html` 파일이 **다음 MvpBuild(6단계) 빌드의 핵심 입력**이다. 브라우저로 열어 화면·인터랙션·토큰을 픽셀 단위로 확인하고, 필요하면 파일 안에 내장된 소스(컴포넌트 로직·CSS)를 직접 참고할 수 있다 — MVP에서 *어떤 환경*(Next.js/Vite · Vue · SwiftUI · 토스 TDS 등)으로도 이 파일을 스펙 삼아 구현한다.

코드에 남지 않는 결정(법적 경계·백엔드 연동 방향 같은 것)은 이 파일에 없다 — 필요하면 MvpBuild 진입 시 그때 사용자에게 다시 확인하고 `mvp-build.md`에 기록한다 (README·DEV-NOTES 같은 별도 문서를 미리 만들어두지 않는다).

ProtoRetro(통과 기준③)의 go/no-go 판정으로 이어진다: go면 같은 스킬의 6단계 MvpBuild가 이 파일을 base로 실서비스 빌드(스택 결정·이식 방식은 6-mvp-build.md §2.2), no-go(다른 아이디어)면 새 프로젝트에서 처음부터.
