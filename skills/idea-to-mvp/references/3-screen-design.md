ScreenDesign는 이 스킬의 **무게중심**이다. AI 빌더 시대엔 빌드가 거의 공짜라 병목이 *생각*이 아니라 *정확한 명세*로 옮겨갔다 — ScreenDesign가 그 명세다.

**이 단계의 산출물은 2개다**:

1. **`screen-design.md`** — *사용자 플로우 + IA·내비 구조(사이트맵·탭바·햄버거·진입 분기) + 전 화면 명세 + 핵심 시나리오*까지 빠짐없이 담는 단일 SoT(진실의 원천).
2. **`wireframes/`** — 그 명세대로 손으로 짠 html/css/js 검증 화면. 플로우·IA가 종이에선 그럴듯해도 막상 클릭해보면 동선이 끊기거나 빠진 화면이 드러난다. 만들어 *직접 클릭하며 점검*하고, 어긋나면 `screen-design.md`를 고치고 다시 만드는 걸 **반복해 최종 확정**한다.

비유로 — `screen-design.md`는 *건물 도면 + 지하철 노선도*, `wireframes/`는 그 도면대로 만들어 *직접 다녀보는 작동 모델*이다. 다녀봐야 "여기 문이 없네", "이 환승 어색하네"가 보인다. **이 단계의 합격선은 *플로우·IA가 막힘없이 도는지* 검증해 확정** — 시각 완성도는 다음 단계(Prototype)가 도맡으니 여기선 시각 X.

**ScreenDesign가 도맡는 차원**:

- **사용자 플로우** — 진입~핵심 작업~끝~재방문 동선
- **IA·내비 구조** — 사이트맵·탭바·햄버거·진입 분기·로그인 상태별 분기
- **전 화면 명세** — 진입조건·보이는 것·행동·상태·데이터·전환 + IA 요소(탭바 유무·hero·CTA 패턴·layout 분기·MVP 마스크 등)
- **핵심 시나리오** — User Story + 화면 경로
- **검증화면** — 위 모든 차원을 클릭으로 검증

다음 4단계 Prototype은 *시각·디자인 시스템*만 도맡음 — 분담 명확.

**카피 톤**: 화면 카피·상태 문구는 market-research §최종 서비스 기획 요약 내용의 *최종 아이디어*가 풍기는 톤 + `ux-writing-reviewer` 에이전트의 토스 메커닉을 따른다 (카피 작성/수정 후 *반드시* 그 에이전트로 점검·교정 — §7.1). 별도 brand 문서는 없다 — 시각·디자인 시스템은 다음 단계 Claude Design이 생성.

**목차**
- §1 단계 목표 (플로우·IA·전 화면 명세·시나리오·검증)
- §2 진행 절차 — §2.2 플로우 골격 · §2.3 화면 명세(7요소) · §2.4 시나리오 경로 · §2.5 IA·내비 구조 · §2.6 검증 화면 빌드·점검(핵심) · §2.7 어휘 swap 절차
- §3 완료 체크리스트 (§3.1 핸드오프 · §3.1.1 카피 sync 절차)
- §4 산출물 스펙 (§4.1 screen-design.md 구조)
- §5 업데이트 모드 · §6 좋은/나쁜 예(§6.6 재사용 패턴 A~G) · §7 응대 톤(§7.1 ux-writing-reviewer · §7.2 핸드오프 안내)

## 1. 단계 목표

- **사용자 플로우** (1) — 처음 진입 → 핵심 작업 → 끝 → 재방문 동선
- **IA·내비 구조** (2) — 사이트맵 텍스트 다이어그램 + 탭바·햄버거·진입 분기·로그인 상태별 분기
- **전 화면 명세** (3) — 모든 화면을 빠짐없이. 각 화면마다 7요소: *IA 요소* + 진입조건·보이는 것·행동+결과·상태·데이터·전환
- **핵심 시나리오 3~5개** (4) — User Story 형식 + 플로우 위 경로 추적 (어떤 시나리오가 어느 화면들을 지나가나)
- **검증 화면 + 플로우·IA 확정** (5) — 위 명세대로 손으로 짠 html/css/js로 클릭하며 점검 → 어긋난 부분 `screen-design.md` 수정 → 다시 빌드 → 반복. **클릭으로 플로우·IA가 막힘 없이 돌 때까지.**

**여기서 다루지 않는 것**: *최종 디자인 프로토타입*을 Claude Design으로 만드는 일 — 그건 다음 **Prototype 단계**다. 3단계 검증 화면을 얼마나 디자인할지는 제약이 없지만, 이 단계의 *목표·합격선*은 **시각 완성도가 아니라 플로우·IA가 맞는지**다.

**SoT 책임** (SKILL.md §2.1 SoT 매트릭스):

- 시나리오 · 플로우 · *IA·내비 구조* · 화면 간 전환 · **화면별 IA 요소·내용·행동·상태(빈·로딩·에러)·데이터**
- `wireframes/`(검증 화면)는 이 SoT를 *점검·확정하는 스캐폴딩*이다 — SoT 아님. 화면과 명세가 어긋나면 **`screen-design.md`가 진실의 원천**이고, 화면을 거기 맞춘다.

Prototype·MVP PRD엔 이 정보를 박지 않고 screen-design 참조로 redirect.

## 2. 진행 절차

### 2.1 단계 시작 — market-research 최종 서비스 기획 요약 내용 base

- `market-research.md` §최종 서비스 기획 요약 내용(최종 아이디어 · 비목표 · 검증 가설)을 base로 인터뷰. **비목표는 화면 범위의 가드레일** — 비목표로 "안 만들 것"은 화면 명세에서 명시 제외한다(부풀음 방지).
- **진입 스냅샷 작성** (SKILL.md §2.3): `screen-design.md` 머리에 `# 진입 스냅샷` 섹션을 박는다 — 직전 스냅샷(market-research §요약)과 대조해 변경 없으면 `변경 없음 — market-research.md §최종 서비스 기획 요약 내용 그대로` 한 줄, 변경 있으면 4개(최종 아이디어·비목표·검증 가설·북극성) 전문 재기술 + `## 변경`에 무엇이 왜. **이 단계 진행 중 방향이 바뀌어도 market-research.md를 고치지 않는다** — 이 스냅샷을 갱신한다.
- 옛 단계 evidence는 SKILL.md §2.3 검토 절차(4 체크 질문 + 라벨)를 거쳐 사용.
- screen-design 산출물 이미 있음(스킬 중간 재실행 등): 내용 읽고 §3 체크리스트 검증, 부족분만 보완. 검증 화면이 이미 있으면 명세와 동기 맞는지 확인.

### 2.2 Step 1. 전체 플로우 골격

`market-research.md` §최종 서비스 기획 요약 내용의 **최종 아이디어**를 사용자에게 다시 보여주고 묻는다. 최종 아이디어가 풍기는 톤을 head에 두고 카피 적용(`ux-writing-reviewer` 에이전트 메커닉).

```
확정한 아이디어가 [최종 아이디어 한 문단]였지. 그 사람이 이걸 처음 만나서
끝까지 쓰는 동안 어떤 순서로 뭘 하는지 골격부터 그려보자.

- 시작점: 어디서 알고 들어와? (진입)
- 처음 5분 안에 뭘 해?
- 핵심 작업이 뭐야? (목적 달성하는 그 순간)
- 끝낼 때 뭘 가지고 나가?
- 다시 돌아와서 또 쓰는 시나리오가 있어?
```

답을 받아 플로우 다이어그램 텍스트(또는 단계 리스트)로 정리해 **사용자 확인 1회** (여기까지 인터뷰 모드 — 질문은 배치로, SKILL.md §1.5). 이 골격 확인이 초안 완주 전 마지막 checkpoint다.

### 2.3 Step 2. 화면 빠짐없이 나열 + 각 화면 명세

플로우 골격에서 **등장하는 화면을 하나도 빠뜨리지 않고** 끌어낸다. **여기서부터 Step 5(검증 화면 빌드)까지는 초안 모드(SKILL.md §1.5)** — 화면 명세 → 시나리오 → IA → 검증 화면까지 한 호흡에 완주하고, 리뷰는 §2.6 클릭 검증에서 한 번에 받는다. 명세 중간 확인 왕복 없음. 불확실 지점은 에스컬레이션 기준 해당 시에만 배치 질문, 그 외엔 합리적 가정 + `추후 결정:` 마킹 후 계속.

각 화면은 다음 **7개를** *전부* 채운다 (하나라도 비면 검증 화면·Prototype에서 추측이 생긴다):

1. **IA 요소** — 탭바 유무·활성 탭 / hero 종류(marketing·압축·없음) / CTA 패턴(인라인·하단 floating·sticky·없음) / layout 분기(상태별 영역 변형 — 한 화면 다 모드 패턴 §6.6.B) / MVP 마스크 영역(있으면 명시)
2. **진입조건** — 어디서 어떤 행동으로 이 화면에 오나
3. **보이는 것** — 주요 요소·콘텐츠. *실제 카피·데이터 필드*까지 (placeholder라도 구체값)
4. **가능한 행동** — 버튼·입력·제스처 각각, 그리고 *각 행동의 결과*(다음 화면 또는 상태 전환)
5. **상태** — 빈 / 로딩 / 에러 / 완료 중 이 화면에 해당하는 것 (없으면 "해당 없음" 명시)
6. **데이터** — 들어오는 필드 / 나가는 필드 (다음 화면·다음 단계로 넘기는 값)
7. **전환** — 이 화면에서 갈 수 있는 모든 다음 화면 (끊긴 곳 없게)

**비목표로 안 만드는 화면**은 "제외: [화면] — market-research §비목표 [사유]"로 명시(존재만 표시할지/아예 뺄지 포함).

### 2.4 Step 3. 핵심 시나리오로 경로 추적

```
이제 가장 중요한 시나리오 3~5개만 골라보자. User Story 형식 +
이 시나리오가 위에서 적은 화면들 중 어디를 지나가는지 경로까지.
"[누구]로서 [무엇]을 [왜]하고 싶다 — 지나는 화면: S1→S3→S5"
```

시나리오가 명세한 화면 위에 실제로 얹히는지 확인 — 지나갈 화면이 명세에 없으면 Step 2로 돌아가 보강.

### 2.5 Step 4. IA·내비 구조 설계

플로우·화면 명세가 한 바퀴 정리되면, *화면 간 구조*를 **사이트맵 텍스트 다이어그램 + 내비 컴포넌트 + 진입 분기 매트릭스**로 박는다. 옛 가이드에선 이 단계가 빠져 *작업 중 자연 박힘에 의존*했는데, 다음 프로젝트 빌더가 *체계적으로* 박게 명시 신설.

**박을 4개**:

1. **사이트맵 텍스트 다이어그램** — 진입~로그인 후~상태별 분기까지 한 장. 예:
   ```
   비로그인 (외부 진입)
     └ LP. 랜딩
         ├ 6단계 카드 [자세히 →]
         │    └ LP-1 ~ LP-6 상세
         └ [시작하기] → signup → 로그인 후

   로그인 후 (탭바 4개)
     ├ 🏠 홈 (4 모드 분기: guest/empty/resident/progress)
     ├ 📦 계약 준비 (S1~L1)
     ├ 🏘 거주 중 (M0·MVP 알림)
     └ ☰ 더 보기 (햄버거 → LP/LP-N/내 활동/설정/도움말/로그아웃)
   ```

2. **내비 컴포넌트 결정**:
   - **탭바** — 노출 여부·탭 개수·각 탭의 진입 대상 (모바일 첫 번째 진입)
   - **햄버거 메뉴** — 그룹별 항목 (서비스 소개·내 활동·설정·도움말·로그아웃 등)
   - **헤더·뒤로가기** — 어느 화면에 [←] 있고 어느 화면에 없는지
   - **하단 floating CTA** — 노출 화면·라벨

3. **진입 분기 매트릭스** — 비로그인 / 로그인 (상태별)을 표로:

   | 진입 상태 | 상단 영역 | 본문 영역 | 탭바 | CTA |
   |---|---|---|---|---|
   | 비로그인 | marketing hero | 6단계 큰 카드·"왜·안 하는 것" | X | 하단 floating |
   | 로그인 + 빈 | 압축 hero | 3 분기 카드 + 6단계 작게 | 4개 | X |
   | 로그인 + 거주 | 지금 사는 집 | 알림(MVP) + 진행없음 + history | 4개 | X |
   | 로그인 + 진행 | 지금 사는 집 + 진행 카드 | 알림(MVP) + history | 4개 | X |

4. **한 화면 다 모드 / 모드 토글 검토** (재사용 패턴 §6.6.B) — *단일 URL + 콘텐츠 자동 분기*가 사용자 *상태 다양성* 자연 흡수에 유리할지 검토. 분리(LP↔홈) vs 통합(같은 URL) 결정. 검증화면에선 *모드 토글 vbar*로 모든 상태 점검 가능하게.

### 2.6 Step 5. 검증 화면 빌드 + 점검 반복 (이 단계의 핵심)

명세(§2.3) + 시나리오(§2.4) + IA(§2.5)가 한 바퀴 정리되면, 그걸 **손으로 짠 html/css/js 화면으로 만들어 클릭하며 플로우·IA를 검증**한다. 종이 위 명세는 빈틈을 숨긴다 — 클릭해봐야 끊긴 전환·빠진 화면·어색한 순서·내비 누락이 드러난다. **여기가 이 단계 리뷰의 본체다** — 초안 모드로 완주한 명세·IA·화면을 사용자가 처음 한 번에 리뷰하는 지점. markdown 명세를 눈으로 훑는 것보다 클릭이 빈틈을 훨씬 잘 드러낸다.

**빌드 원칙** (Prototype 단계와 구분되는 *검증용* — 합격선은 시각이 아니라 플로우·IA):

- **AI가 화면을 직접 빌드** — `screen-design.md` 명세를 그대로 옮긴 html/css/js. 화면마다 보이는 것·행동·전환·상태를 클릭으로 따라갈 수 있게.
- **전환이 실제로 동작** — 버튼·링크 누르면 명세한 다음 화면으로 이동. 핵심 시나리오 경로를 끝까지 클릭으로 통과 가능.
- **상태도 눈에 보이게** — 빈·로딩·에러·완료 상태를 토글·예시 화면 등으로 확인 (명세에 있으면).
- **데이터는 placeholder 고정** — 명세 데이터 필드 + 최종 서비스 기획 요약 내용 첫 베타 케이스로 구체값.
- **IA·내비·구조·layout 분기는 박는다** — 탭바·햄버거·floating CTA·모드 토글·MVP 마스크 다 박음. *시각만 다음 단계*. 이걸 빼면 *구조 결정*까지 4단계(Prototype) Claude Design에 떠넘기게 되어 분담 모호.
- **시각은 *와이어프레임 톤으로 제약* (강제)** — 검증화면은 *플로우·IA 비계(scaffolding — 임시 발판)*. 시각이 박힌 검증화면은 *layout anchor*로 작용해 Claude Design이 외형을 베끼게 만든다 (4-prototype.md 첫머리 "이미 렌더된 시각이 layout anchor로 작용" 위험과 동일 메커니즘).
  - **허용**:
    - 흑백·회색 단색 / 시스템 sans-serif / 1px 회색 테두리 박스
    - 단순 상태 토글·점프 메뉴 / 화면 간 전환
    - **IA 구조 (탭바·햄버거·floating CTA·모드 토글)**
    - **MVP 마스크 (반투명 + 비활성 + 별도 "MVP 예정" 안내)** (재사용 패턴 §6.6.A)
    - **검토용 보조 도구** (devbar 점프 select·모드 토글 vbar) — *앱 아님 명시*, 재사용 패턴 §6.6.E
  - **금지** (베낄 단서가 생김):
    - 그라데이션·box-shadow·정교한 radius
    - brand color 2개 이상 (강조는 굵게·박스로)
    - 아이콘·이모지로 *의미 전달* (🔔·⚠️·✅ 같은 것 — 텍스트 라벨 `[알림]`·`[경고]`·`[OK]`로 대체)
    - 이미지·일러스트·hero 영역(사진·그림)
    - 디바이스 chrome (폰 프레임·notch 같은 외형)
    - hover·focus 디테일 / 다크모드 / 반응형 정교화
  - **이유**: 합격선은 *플로우·IA가 맞는지*이지 시각 완성도가 아님. 와이어프레임 톤이면 (1) 사용자가 "예쁘게"에 시간 안 뺏기고 (2) `ux-writing-reviewer`는 카피만 보니 영향 없고 (3) 다음 단계 Claude Design이 시각을 *새로 자유 결정*. *최종* 디자인 프로토타입은 다음 Prototype 단계에서 Claude Design이 만든다.
- **비목표 화면은 안 만든다** — market-research §비목표 제외 화면은 검증 화면에도 없음.

**검증 루프** (`CLAUDE.md` "검증" 룰: AI가 Playwright 등 자동 도구로 검증하지 않음 — 사람이 브라우저로 클릭):

1. AI가 `wireframes/`에 화면 빌드.
2. **사용자가 브라우저로 열어 핵심 시나리오대로 클릭** → 막히는 곳·어색한 순서·빠진 화면·내비 누락 피드백.
3. AI가 피드백을 **먼저 `screen-design.md`에 반영**(SoT 갱신) → 화면 재빌드.
4. 클릭으로 플로우·IA가 막힘 없이 돌 때까지 2~3 반복. **최종 확정.**

**동기화 불변식 (변경 종류별로 다른 시점)** — 검증화면이 *살아있는 작업 도구*고 `screen-design.md`가 *최종 SoT*:

| 변경 종류 | SoT | 동기 시점 |
|---|---|---|
| **구조 변경** — 화면 추가·삭제·순서 변경·전환·상태·데이터 의존성·비목표·IA·시나리오 anchor | `screen-design.md` (즉시) | **검증화면에서 구조 영향 변경을 했으면 *같은 턴에* `screen-design.md` 먼저 반영 후 화면 갱신** |
| **카피 변경** — 헤딩·서브·라벨·버튼·에러·체크 항목 등 텍스트 | 검증화면 (iteration 중) → `screen-design.md` (단계 종료 시 일괄) | iteration 동안엔 검증화면이 *카피 working SoT*. **§3 통과 직전에 AI가 카피를 `screen-design.md`에 일괄 sync** (§3.1.1) |
| **순수 시각 디테일** (CSS 픽셀 조정 등) | 검증화면 only | 동기 불필요 — 시각은 다음 Prototype 단계에서 Claude Design이 새로 결정 |

**왜 카피만 *지연 동기*인가**: (1) iteration 중 카피를 *매번* `screen-design.md`까지 옮기면 노이즈 ↑·작업 흐름 끊김. (2) 검증화면이 카피의 *자연스러운 working 공간* — 클릭으로 톤 보면서 다듬고 `ux-writing-reviewer`도 거기서 돔. (3) 단계 종료 시점에 *한 번에* merge해 단일 SoT로 정리하면 Claude Design 핸드오프 input이 깔끔.

**iteration 중 stale 주의 (강제)**: 검증화면 카피가 다듬어진 뒤 `screen-design.md` sync 전까지 — `screen-design.md` "보이는 것" 항목의 카피는 *옛 안*이라 의사결정 기준으로 쓰면 안 됨. 사용자나 AI가 그 사이 카피 의사결정을 해야 하면 *검증화면을 본다*. (§3 통과로 자동 정리됨)

### 2.7 어휘 swap·이름 정합 절차

작업 중 핵심 어휘·단계명·산출물명을 swap해야 할 때(이번 프로젝트 예: 카탈로그→모음 / 질문 스크립트→물어볼 말 / UserFlow→ScreenDesign), 일관 정리 절차:

1. **swap 범위 파악** — `grep -rn "<옛 단어>" .claude/ planning/`로 모든 흔적 list.
2. **순서대로 sed 일괄** — 긴 패턴 먼저(예: `userflow-screens` → `wireframes`), 짧은 패턴 나중(예: `userflow` → `screen-design`).
3. **파일·폴더 rename** — 텍스트 swap 후 `mv`로 정합.
4. **anchor 인용 *원문 보존*** — *다른 SoT를 직접 인용*하는 anchor는 원문 그대로 두고 footer에 표기 (anchor 정합 우선). 인용 원본(market-research 등)을 같이 swap할 때만 anchor도 swap.
5. **검증** — 다시 `grep`로 잔재 0 확인.
6. **🟥 폐기 footer 박기** — 옛 어휘·가정 폐기 명시 + 새 어휘 + 사유 + 영향 범위.

재사용 패턴 §6.6.F 참조.

2. **최신 진입 스냅샷** — 최종 아이디어·비목표·검증 가설·북극성. *무엇·왜·누구·스코프·톤*. `screen-design.md` 머리 스냅샷이 `변경 없음`이면 `planning/mvp/market-research.md` §최종 서비스 기획 요약 내용이 그 원문 (SKILL.md §2.3).

> **🔁 검증화면(`wireframes/`)은 input으로 *들고 가지 않는다***. 검증화면은 *3단계(ScreenDesign) 안 작업 비계* — 플로우·IA 클릭 검증 + 카피 다듬기 도구로 쓰이고, 핸드오프 직전에 **카피가 `screen-design.md`로 일괄 sync**된다(§3.1.1). 검증화면 html을 input으로 주면 DOM 구조가 layout anchor로 작용해 Claude Design 자유 재설계를 묶을 위험. *단일 SoT인 screen-design.md로 통합*이 깔끔.

검증화면 자체는 핸드오프 후에도 **산출물 폴더에 보존** — retro 시 *클릭 가능한 비계*로 다시 보기 + git history로 진화 추적 + 다음 프로젝트 형식 참고.

#### 3.1.1 검증화면 → `screen-design.md` 카피 sync 절차 (체크리스트 마지막 항목, AI 1회 자동)

§3 체크리스트의 *플로우·IA 검증 완료* 이후, **카피 sync 완료** 항목을 채우기 위해 AI가 자동으로:

1. **추출**: `wireframes/index.html`을 읽고 화면별 카피를 모음 — 헤딩·서브·섹션 라벨·체크 항목·버튼·에러 메시지 등. (시각·CSS·layout 정보는 안 가져옴.)
2. **비교**: 같은 화면의 `screen-design.md` "보이는 것" 항목 안 인용 카피(`"…"` 표기된 부분)와 *문구별 diff* 생성.
3. **사용자 확인**: 변경된 카피를 사용자에게 *diff 표*로 보여줌 — `before → after`. 인용 부호 밖의 *추상 묘사*("까다롭다 소리 안 듣는 톤" 같은)는 *건드리지 않음* (사람 결정 영역). 묘사가 새 카피와 어긋난다 싶으면 별도 보고.
4. **갱신**: 사용자 확인 후 `screen-design.md` "보이는 것" 항목의 *인용 카피만 정밀 교체*. 묘사·구조·전환 명세는 그대로.
5. **체크리스트 마킹**: 체크리스트의 *카피 sync 완료* 항목 체크.

이후 `screen-design.md`가 *플로우·IA·구조 SoT + 최신 카피*를 모두 담은 단일 SoT가 되어 4단계(Prototype) Claude Design 핸드오프 input으로 그대로 들어감.

**받아 와야 할 output — zip 핸드오프 패키지** (4-prototype.md §1·§4 산출물):

- Claude Design이 프로토타입을 거의 완성해 **zip 하나**로 내보낸다. 안에는 진입 **reference HTML**(브라우저로 더블클릭하면 도는 *작동 데모이자 시각 정답지* — 파일명 가변, README가 진입 파일 지정) + **README**(화면·토큰·동작 설계도 + 이식·TDS 매핑) + `src/tokens/`·컴포넌트 소스(MVP 빌드용)가 들어 있다. (`design-system.html`은 있을 수도, 없을 수도.)

받아 온 **zip은 Claude Code(여기)에서 `planning/mvp/prototype-ver-1/`에 압축 해제 → 4단계(Prototype) 본격 시작** (4-prototype.md §2.1).

## 5. 재실행 시 처리

이 스킬은 프로젝트당 1회 — 별도 재실행 시 처리가 없다. ProtoRetro(통과 기준③)의 go/no-go 판정으로 이어진다: go면 같은 스킬의 6단계 MvpBuild로 계속, no-go(다른 아이디어)면 새 프로젝트에서 처음부터. (`references/5-proto-retro.md` §2.7.)

## 6. 좋은 예 vs 나쁜 예

핵심 차이는 **플로우·IA·화면 명세 빠짐없이 + 클릭으로 검증**(좋은) vs **추상 흐름만·IA 누락·화면/상태 누락·검증 안 함**(나쁜).

### 6.1 사용자 플로우

**좋은 예** (진입~재방문 구체 + 화면 ID — Airbnb 예):
> [검색 진입] → S1 검색 → S2 매물 목록 → S3 매물 상세 → S4 예약·결제 → [완료]. 재방문: 예약 알림 → S5 여행 관리.

**나쁜 예** ← 추상·검증 불가:
> 사용자가 앱에 들어와 사용하고 만족하며 나간다.

### 6.2 IA·사이트맵

**좋은 예** (텍스트 다이어그램 + 진입 분기 + 내비 명시):
> ```
> [비로그인]
>   └ LP. 랜딩 → [시작하기] → signup → 로그인 후
> [로그인 후 — 탭바 4개]
>   ├ 🏠 홈 (4 모드: guest/empty/resident/progress)
>   ├ 📦 계약 준비
>   ├ 🏘 거주 중
>   └ ☰ 더 보기 (햄버거)
> ```
> + 진입 분기 매트릭스 4행(비로그인/empty/resident/progress) × 4열(상단/본문/탭바/CTA) + 내비 컴포넌트 4종 명시.

**나쁜 예** ← 화면 list만 있고 *연결·진입 분기·내비*가 없음:
> 화면 12개 있다.

### 6.3 화면 명세

**좋은 예** (7요소 다 채움, IA 요소 명확 — Airbnb 예):
> ## S3. 매물 상세
> - **IA 요소**: 탭바(검색 on) / hero 없음 / CTA(하단 sticky "예약하기") / 상태 토글(빈/로딩/에러)
> - 진입조건: S2 목록에서 매물 카드 탭
> - 보이는 것: 사진 캐러셀, 제목·가격(₩120,000/박), 후기 평점 4.9, 호스트 정보, 날짜 선택
> - 가능한 행동: [날짜 선택] → 가격 갱신 / [예약하기] → S4 / [뒤로] → S2
> - 상태: 빈 = "이 날짜엔 자리가 없어" + [다른 날짜] / 로딩 = "예약 가능 확인 중" / 에러 = "잠깐 안 불러와져, 다시"
> - 데이터: 입력 = 매물 1건·선택 날짜 / 출력 = 예약 정보(S4 결제 input)
> - 전환: → S4, → S2

**나쁜 예** ← 화면 통째 누락 + IA·상태·전환 없음:
> 상세 화면에서 정보를 보여준다.

### 6.4 핵심 시나리오

**좋은 예** (페르소나·경로·측정값):
> 여행자로서, 주말 숙소를 고를 때 사진·후기로 빨리 판단하고 싶다. 지나는 화면: S2 → S3. 기대결과: 5분 내 "여기 예약할까" 결정.

**나쁜 예** ← 추상:
> 사용자로서 편하게 쓰고 싶다.

### 6.5 검증 화면

**좋은 예** (와이어프레임 톤 + IA 박힘 + 모드 토글 + 시나리오 끝까지 클릭):
> S1~S5를 흑백·회색·sans-serif 박스. [예약하기] 누르면 진짜 S4로. 빈/로딩/에러는 `[로딩]`·`[에러]` 텍스트 라벨로 토글. **탭바 4개·하단 floating CTA·햄버거 메뉴 박음**. 거주 중 알림은 MVP 마스크(반투명+비활성+"MVP 예정" 안내). 검토용 *모드 토글 vbar*로 4 진입 상태 점검. 클릭해보니 S3 뒤로 갈 곳이 없어서 명세에 [뒤로]→S2 추가하고 화면도 고침. (시각 결정 0 — 다음 단계 Claude Design 자유.)

**나쁜 예** ← 전환 죽음 / 명세 따로 / 시각 박힘 / IA 빠짐:
> 버튼 눌러도 화면 안 바뀜. screen-design.md엔 없는 화면이 검증화면에 있음. 그라데이션·아이콘·brand color 다듬느라 "이 순서 맞나" 한 번도 클릭 안 함. 탭바·내비 빠짐 — 다음 단계 Claude Design이 *IA·시각 둘 다* 새로 정해야 해서 분담 모호.

### 6.6 재사용 패턴

다음 패턴들은 다양한 prototype에서 재사용 가능. 집내비 프로젝트 진행 중 자연 박혔고, 명시 박아두면 다음 빌더가 의식적으로 활용.

#### A. MVP 마스크 패턴

**언제**: prototype에서 *작동은 비싸지만 화면 명세는 필요한 기능* (자동 모니터링·알림·서드파티 API·결제·인증 등).

**어떻게**: 화면 영역을 **반투명**(CSS `opacity: 0.35`) + **비활성**(`pointer-events: none; user-select: none`) + **별도 안내 박스**에 *"MVP 예정"*·*"prototype에선 화면만"* 명시.

**예**: 집내비 거주 중 자동 알림(등기 변동·HUG·임대인 변경) — 모니터링 인프라 빌드는 MVP에서, prototype에선 *화면 명세만* 마스크 처리. 회원가입 stub도 같은 패턴.

#### B. 한 화면 다 모드 (단일 URL + 콘텐츠 자동 분기)

**언제**: 사용자 *진입 상태가 다양*하고(비로그인 / 로그인 + 여러 상태), URL을 *하나*로 통합하는 게 자연스러운 경우. SaaS 표준 (Linear·Notion·Figma `/` 라우트).

**어떻게**: URL 하나, *상태에 따라 상단·본문 영역만 자동 토글*. 공통 영역(6단계 카드·교육 자료 등)은 그대로. 검증화면에선 *모드 토글 vbar*로 모든 상태 점검.

**예**: 집내비 홈 — 비로그인 = LP / 로그인 = 사용자 상태 카드. 단일 `/`. 옛 *LP·홈 분리*를 *통합 + 모드 토글*로 통합.

**언제 분리**: 콘텐츠·tone 차이가 너무 크거나 SEO·마케팅·앱 IA 따로 운영 필요 시. 결정은 §2.5 Step 4.

#### C. "우리가 안 하는 것" 섹션

**언제**: 제품 범위가 *경쟁 도구와 헷갈릴 위험*이 있을 때. 사용자가 *우리가 그것도 한다고 오해*할 가능성.

**어떻게**: LP에 명시 — *우리가 안 하는 것* + 대체 도구 안내.

**예**: 집내비 LP — "매물 검색·중개 (직방·다방) / 시세·실거래가 (호갱노노)" 명시.

**효과**: 사용자 기대 정합 + 제품 부풀음 방지.

#### D. lifecycle 모델 (단발 → 평생 여러 인스턴스)

**언제**: 제품이 *반복 발생*하는 사용자 행동(계약·이사·이력서·여행·연말정산 등)을 다룰 때.

**어떻게**: *허브 + 인스턴스* 패턴 — 허브가 *전체 history + 현재 진행 + 다음 예정*을 관리, 각 인스턴스가 *한 사이클*의 작업 흐름.

**예**: 집내비 — 주거 lifecycle 허브 + 각 계약 인스턴스(S1~L1 6단계 워크플로우 한 사이클).

#### E. 검토용 보조 도구 (devbar · 모드 토글 vbar)

**언제**: 검증화면에서 *모든 화면·상태를 한 번에 점검* 필요 시.

**어떻게**: 화면 위 또는 안에 *앱 아님 명시*의 점프 select·모드 토글·상태 토글 박음. 사용자가 클릭 한 번으로 검증 가능. 시각은 와이어프레임 톤 그대로.

**예**: 집내비 검증화면 devbar(점프 select — 12 화면 + 4 모드) + home 안 모드 토글 vbar.

#### G. 상태 기반 동적 floating CTA

**언제**: 한 화면이 *행위(send/submit/diagnose/register/pay)* + *시퀀스(다음 단계)*를 함께 가지는 경우. 결제·전송·등록·진단 화면.

**문제**: 같은 화면에 [행위 버튼](예: 카톡으로 보내기) + [다음 단계 버튼](예: 다음 화면 →) 둘을 *동시 노출*하면:
1. **Primary Action 1개 원칙 위배** (Apple HIG·iOS·Material 정석)
2. **Visual Hierarchy 혼란** — *행위*가 진짜 메인인데 *다음 단계*가 동등하게 보임
3. **Sequential Logic 약화** — *행위 전*에 다음 단계로 가는 길이 열려있음 = 자연 흐름 어긋남

**어떻게**: floating CTA가 *현재 상태(`data-state`)에 따라 자동 변함*. 단일 primary 유지.

```
draft (행위 전):  floating = [행위 버튼]
sent  (행위 완료): floating = [다음 단계 →]
error (행위 실패): floating = [다시 시도]
```

CSS 패턴 (state별 자동 토글):
```css
#screen .state-draft, #screen .state-sent, #screen .state-error{display:none !important;}
#screen[data-state="draft"] .state-draft{display:revert !important;}
#screen[data-state="sent"] .state-sent{display:revert !important;}
#screen[data-state="error"] .state-error{display:revert !important;}
```

HTML 패턴 (floating CTA 안 상태별 div):
```html
<div class="floating-cta">
  <div class="state-draft"><button class="btn primary" onclick="setState('sent')">행위 버튼</button></div>
  <div class="state-sent"><button class="btn primary" onclick="go('next')">다음 단계 →</button></div>
  <div class="state-error"><button class="btn primary" onclick="setState('sent')">다시 시도</button></div>
</div>
```

**왜 좋은가**:
- **Primary Action 1개 원칙** — 한 화면 = 1 primary. 위계 명확
- **순서 강제** — 행위 *전*엔 다음 단계 못 감. 자연 흐름
- **상태 변화 = 흐름 자연** — 사용자 다음 액션 *항상 1개*
- **iOS·Material 정석** — 익숙한 인터랙션 (Apple Pay·iMessage·Slack 등 패턴 동일)

**다중 상태 케이스 (3+)**: input → loading → result 같은 진단·결제 흐름엔 상태 ↑:

```
input:   floating = [진단받기]
loading: floating = [진단 중...] disabled (opacity:0.5)
result-a: floating = [다음 →] + 보조 [되돌리기]
result-b: floating = [다음 →] + 보조 [되돌리기]
```

**보조 액션(secondary)** 처리: floating CTA 안 *별도 div*로. primary 아래 [btn ghost btn-group]으로 *secondary 2개 그룹*.

**검증화면 패턴**: 상태 토글 vbar(검토용 vbtn)로 모든 상태 직접 점검 가능. devbar select에서 점프해도 *default state*로 진입 — *다른 state*는 vbar로 토글.

**언제 분리 (현재 패턴 유지)**: 행위가 *완전 독립*적이고 *다음 단계 진행과 무관*한 경우 (예: 검색 버튼 + 도움말 진입). 그 외는 동적 floating이 정석.

#### F. 어휘·단계명·SoT·anchor 정합

**언제**: 어휘·단계명·산출물명 swap 시 (예: 카탈로그→모음 / UserFlow→ScreenDesign).

**어떻게**: §2.7 절차 — `grep`으로 범위 파악 → 긴 패턴 먼저 `sed` → 파일·폴더 rename → anchor 인용 *원문 보존* → 잔재 검증 → 🟥 폐기 footer.

**예**: 집내비 프로젝트에서 *UserFlow → ScreenDesign* swap — references·산출물·폴더 일괄 sed + rename + anchor 인용은 원문 보존(market-research 본문 갱신 시 동기 sync).

## 7. 사용자 응대 톤 + 인터뷰 코칭

- **톤**: SKILL.md §1.3대로 반말·친근·짧게. 골격은 인터뷰 모드(배치 질문)로 확인 1회 → 이후 명세·시나리오·IA·검증 화면은 초안 모드로 완주해 한 번에 낸다 (§2.3).
- **AI가 먼저 화면 명세 초안** (SKILL.md §1.3): 플로우에서 화면을 빠짐없이 끌어내 *7요소* 초안까지 같은 턴에 낸다. 사용자는 빠진 화면·틀린 전환만 짚어주면 됨.
- **AI가 먼저 IA 다이어그램 초안**: 화면 명세 한 바퀴 정리되면 *사이트맵 텍스트 다이어그램 + 진입 분기 매트릭스 + 내비 컴포넌트* 초안 같은 턴에 낸다.
- **검증 화면도 AI가 먼저 빌드**: 명세·IA 정리되면 "이제 손으로 화면 만들어서 클릭해볼 수 있게 해줄게 — 브라우저로 열어 시나리오대로 눌러보고 막히는 데 알려줘" 하고 같은 흐름으로 빌드까지 낸다. 자동 검증 X(사람이 클릭) — `CLAUDE.md` "검증" 룰.
- **코칭**: SKILL.md §1.4대로. 화면 명세가 §6 좋은 예(7요소)에 못 미치면 빠진 요소 짚어 되묻기 ("이 화면 에러 상태는? 여기서 어디로 가? 탭바 어디 활성?"). "잘 모르겠어"엔 후보 제시. 모르는 부분은 `추후 결정: ...`로.
- **누락 사냥이 핵심 동작**: 플로우·IA·시나리오에 언급됐는데 명세 안 된 화면, 전환이 끊긴 화면, 상태가 빠진 화면, 내비 누락을 능동적으로 찾아 채운다. **검증 화면을 클릭해 그 빈틈을 실제로 드러낸다.** 빌더의 다음 단계(Prototype 디자인)가 정확하려면 여기서 플로우·IA 빈틈이 0이어야 한다.
- **피드백은 명세 먼저, 화면 나중**: 클릭 검증에서 문제를 찾으면 화면만 고치지 말고 **`screen-design.md`(SoT)를 먼저 고친 뒤** 화면을 거기 맞춘다 — 둘이 어긋난 채 다음 단계로 넘어가지 않게.

### 7.1 ux-writing-reviewer 자동 호출

UI 카피 작성·swap·재구성 *직후*마다 `ux-writing-reviewer` 에이전트를 *자동 호출*해 토스 톤 + 프로젝트 톤 정합을 점검·교정. 진행 동안 카피가 진화하는데 매번 사람이 점검 안 해도 되게.

**언제 자동 호출**:
- 검증화면(`wireframes/index.html`) 통째 작성·재구성 직후
- LP·홈·LP-N 화면 카피 어휘 일괄 swap 직후 (예: 카탈로그→모음)
- 6단계 상세 페이지(LP-N) 4섹션(왜·무엇·어떻게·예시) 카피 작성 직후
- 회원가입·signup stub 카피 작성 직후

**프롬프트 핵심**:
- 프로젝트 컨텍스트 (타겟·서비스명·이번 프로젝트 어휘 결정)
- 검토 대상 파일 path
- 와이어프레임 톤 가드 — *카피만*, 시각·CSS·구조 안 건드림 강조
- 옛 구조(다음 단계에서 재작성 예정)는 *카피만* 손보고 *구조·명세*는 건드리지 마

**받는 결과**: 표 (file:line / before → after / 위반 원칙) + *수정함* vs *사람 결정* 분리.

**검증**: 호출 후 `grep`으로 옛 어휘 잔재 확인. *옛 구조 명세*의 anchor 인용은 *원문 보존* 정책으로 swap 안 된 경우 footer에 표기 (§2.7).

### 7.2 단계 종료 시 다음 단계 핸드오프 안내 (강제)

체크리스트(§3) 통과 *직전*에 **카피 sync 먼저** 한 뒤 멘트 — §3.1.1 절차:

1. **카피 sync 절차**: AI가 `wireframes/index.html` 읽고 → `screen-design.md` "보이는 것" 인용 카피와 *문구별 diff* 작성 → 사용자에게 보여줌 → 확인 후 *인용 카피만 정밀 교체*. (묘사·구조 등은 안 건드림)
2. **체크리스트 *카피 sync 완료* 항목 마킹**.
3. **안내 멘트**:
   > *"플로우 + IA + 카피 sync까지 끝! 이제 외부 Claude Design 갈 차례야.*
