idea-validation은 *문제 강도(painkiller)*만 검증한다 — *같은 문제를 이미 푸는 경쟁자가 있는지*는 못 본다. 진통제 신호가 있어도 강한 경쟁자가 같은 문제를 이미 잘 풀고 있으면, 똑같이 들어가면 진다. 이 단계가 **두 번째 통과 기준 — 경쟁 강도** 검증이다: **비슷한 서비스가 이미 있는지 찾고 → 있으면 얼마나 센지 판정하고 → 너무 세면 아이디어를 고치거나(피봇) 접는다(폐기).**

**목차**
- §1 단계 목표 · §2 진행 절차(§2.2 경쟁자 발견 · §2.3 깊이 분석 · §2.4 경쟁 강도 판정 · §2.5 카테고리·차별화 axis · §2.6 YC 4명 시뮬 · §2.7 최종 서비스 기획 요약(가드레일) · §2.8 "매우 강함" 분기)
- §3 완료 체크리스트 · §4 산출물 스펙 · §5 업데이트 모드 · §6 좋은/나쁜 예 · §7 응대 톤

## 1. 단계 목표

- 진통제 검증 후 **경쟁 강도** 판정 (두 번째 통과 기준 — 첫째: idea-validation painkiller)
- 경쟁자가 이미 문제를 잘 풀고 있으면 **카테고리 재정의 또는 피봇·폐기 결정**
- **끝에 빌더 가드레일 확정** — 경쟁 강도를 판정한 직후 *최종 아이디어 한 문단 + 안 만들 것(비목표) + 검증 가설 + 북극성 1줄*을 박는다. 이게 ScreenDesign·Prototype·Retro 전 단계의 입력.

**왜 이 단계가 필요한가** — Stage 1 인터뷰만으로는 시장 전체 지형을 못 본다 (표본 N=3~5라 이미 있는 강한 도구를 놓치기 쉽다). **이미 같은 진통제를 잘 푸는 강한 경쟁자가 있으면, 똑같은 해결책의 유사 서비스를 또 만드는 건 낭비다** — 그래서 빌드 전에 경쟁 강도부터 확인한다.

## 2. 진행 절차

### 2.1 단계 시작 — Stage 1 산출물 base

- `planning/mvp/idea-validation.md`가 `status: confirmed`(go 분기) + `# 최종 확정 아이디어` 채워졌으면 진입
- 미완료면 멈춤 → 사용자에게 _idea-validation 마무리 안내_. 강행 시 SKILL.md §4 단계 진입 평가 정책 적용

### 2.2 Step 1. 경쟁자 발견 — 사용자 + Agent deep research

**소스 둘**:

1. **사용자 인터뷰에서 들은 경쟁자** — `idea-validation.md` 인터뷰 노트의 _대안 도구·임시방편 언급_
2. **AI가 Agent tool로 deep research 위임** — 사용자가 모르는 경쟁자 발견

**0번 경쟁자 = 사용자의 현재 해결책 (필수 포함)** — 경쟁자는 같은 업종 서비스만이 아니다. 사용자가 지금 문제를 때우는 방법(엑셀·카톡 메모·네이버 카페 검색·"그냥 참기")이 실전에서 가장 센 경쟁자인 경우가 많다. Stage 1 인터뷰가 수집한 임시방편을 **경쟁자 표의 0번 줄**에 넣는다 — 우리 서비스가 이겨야 할 상대는 멋진 경쟁 앱이 아니라 "지금 쓰는 그 엑셀"일 수 있다.

#### Agent 호출 명세

- **agent type**: `general-purpose`
- **호출 시점**: Step 1 시작 시 AI가 자동 호출 (사용자 확인 받고 진행)
- **prompt 형식**:

```
다음 아이디어의 경쟁자를 한국·전세계 시장에서 N=3~7개 찾아 분석해줘.

각 경쟁자별 다음 항목:
- 한 줄 요약 (서비스 정의)
- 코어 기능 리스트
- 입력 방식·결과 형태
- 가격 모델 (무료·프리미엄·B2B)
- 타겟 사용자
- UX 인상 (직관·복잡)
- **약점·빈틈** (서비스가 못 해주는 것 — 가장 중요)

아이디어: [`idea-validation.md` §1.8.1 최종 확정 아이디어 한 줄]
도메인 키워드: [도메인 키워드 3~5개 — 예: "일정 관리", "팀 협업", "노트 앱"]

WebSearch·WebFetch 활용. 결과는 markdown 표로 정리해 반환.
직접 도구가 없으면 *블로그·법무사 글·언론 콘텐츠* 수준이라고 명시.
```

#### 호출 결과 처리

- Agent 결과를 사용자에게 보여주고 _false positive(경쟁자 아닌데 잘못 잡힌 것) 제거·추가 경쟁자 보강_ 받음
- 사용자가 *경쟁자 없다*고 우길 때: AI가 한 번 더 _검증 권장_ (자기 검열) — 진짜 없을 수도 있고 못 찾은 것일 수도

### 2.3 Step 2. 경쟁자 깊이 분석 — 각 경쟁자 WebFetch 보강

Step 1 결과 + 사용자 추가 경쟁자에 대해:

- 각 경쟁자별 WebFetch로 상세 페이지 확인 (가격·기능·UX)
- *직접 접속 가능*한 경쟁자만 (앱·인증 필요 서비스는 사용자가 알려준 내용 기반)
- §4.1 산출물 스펙 표 형식으로 정리

### 2.4 Step 3. 경쟁 강도 판정 (등급 4개 — 텍스트 표기)

| 등급          | 설명                                           | 다음 행동                               |
| ------------- | ---------------------------------------------- | --------------------------------------- |
| **없음**      | 직접 경쟁 서비스가 없다 (블로그 글·법무사 글·뉴스 수준만) | **§2.4.1 의심 절차 먼저** → 통과 시 §2.7 진행 |
| **약함**      | 경쟁 서비스가 있지만 일부 기능만 하거나, UX가 약하거나, 사용자가 적다 | 차별화 axis 1~2개로 충분. Step 4 가볍게 |
| **강함**      | 제대로 된 서비스 + 시장 점유 + 충성 사용자가 있다 | **Step 4·5 필수**                       |
| **매우 강함** | 기존 강자(incumbent)가 자본·브랜드·네트워크 등 해자로 시장을 쥐고 있다 | **피봇 또는 폐기 권장** (§2.8) |

**판정 근거에 포함해야 할 것**:

- 시장 점유 수치 (트래픽·다운로드·언론 노출 등 추정 가능 데이터)
- 충성 고객 신호 (커뮤니티 후기·재사용·추천(referral))
- **해자 체크리스트 5개** — "강함"과 "매우 강함"을 가를 때 아래 5개를 반드시 훑고 결과를 기록한다. 해자는 성 둘레에 파 놓은 물길이다 — 후발주자가 쉽게 못 건너는 장벽. ("몇 개 이상이면 매우 강함" 같은 기계적 기준은 두지 않는다 — 근거 없는 숫자는 가짜 정밀함이다. 5개를 다 훑었다는 기록만 요구.)

| 해자 | 확인 질문 |
|---|---|
| 네트워크 효과 | 사용자가 늘수록 그 서비스가 더 좋아지는 구조인가? (양면 시장·커뮤니티) |
| 전환 비용 | 사용자가 갈아타려면 무엇을 잃는가? (쌓아둔 데이터·익힌 습관·연동) |
| 데이터 축적 | 경쟁자가 시간을 들여 쌓은 데이터를 후발주자가 따라잡을 수 있는가? (schlep — 하기 싫지만 쌓이면 해자가 되는 고된 노동) |
| 브랜드·신뢰 | 이 분야에서는 신뢰가 구매를 결정하는가? |
| 규제·독점 계약 | 면허·라이선스·독점 공급 계약이 진입을 막는가? |

#### 2.4.1 등급 "없음"일 때 — 기뻐하지 말고 의심하라 (필수 절차)

경쟁자가 하나도 없다는 것은 좋은 신호가 아닐 때가 훨씬 많다. **경쟁자가 없다는 건, 그 시장에 돈 낼 사람이 없다는 뜻일 가능성부터 의심해야 한다.** 목 좋은 길목인데 가게가 하나도 없다면, "내가 처음 발견했다!"보다 "여기 장사가 안 되는 이유가 있나?"를 먼저 묻는 것과 같다.

아무도 안 만든 이유는 보통 셋 중 하나다:

1. **사람들이 그 문제 해결을 안 원한다** (시장 없음 — 제일 흔함)
2. **만들어도 돈이 안 된다** (돈 낼 사람이 없다)
3. **정말로 새 기회다** (제일 드묾 — 이 경우엔 "왜 하필 지금 가능해졌나(why now — 기술·규제·행동 변화)"에 답할 수 있어야 한다)

**절차**: AI가 사용자에게 "왜 아무도 안 만들었을까?"를 묻고, 셋 중 어디에 해당하는지 함께 판단한다. 동시에 Stage 1의 진통제 근거(임시방편 보유 신호)를 한 번 더 점검한다 — 임시방편 신호가 강하면 3번(새 기회)의 근거가 되고, 약하면 1번(시장 없음)을 의심한다. 판단 결과를 산출물의 "경쟁 강도 판정" 아래 기록한다.

**반대 방향도 기억하라**: 경쟁자가 있다는 건 그 문제에 돈과 시간을 쓰는 사람들이 실제로 있다는 증거다. 그래서 "약함·강함 + 명확한 차별화"가 "없음"보다 오히려 좋은 패일 수 있다.

### 2.5 Step 4. 카테고리 재정의·차별화 axis 검토 (강함·매우 강함 시)

**9개 차별화 축(axis) 후보**:

| 축(axis)                  | 설명                                                       |
| ------------------------- | ---------------------------------------------------------- |
| **A. 좁은 타겟**          | 고객 세그먼트 1개에 집중 (예: 특정 직군·상황 하나)         |
| **B. 통합 쐐기(wedge)**   | 여러 진통제를 한 흐름에 묶음 (예: 검색 + 비교 + 결제)      |
| **C. 사용 시점 이동**     | 경쟁자가 안 가는 시점으로 (작업 전·후 등)                  |
| **D. 유통 쐐기(distribution wedge)** | 콘텐츠·커뮤니티·검색으로 진입 (SEO·블로그·유튜브) |
| **E. 가이드 톤**          | 친구·멘토 톤 (점수 X → 다음 액션 안내)                     |
| **F. B2B 변환**           | 개인 대상 → 기업 대상(B2B)                                 |
| **G. 데이터 확장**        | 새 데이터 소스 추가 (공공·후기·암묵지)                     |
| **H. 가격 모델**          | 무료·프리미엄·구독(subscription) 등으로 차별               |
| **I. 카테고리 재정의**    | _흐름 역행·정의 자체 변경_ (예: 호텔 예약 → 남의 집 묵기 = Airbnb) |

각 axis × **사이드 모드 적합도** + **YC 평가** 두 점수 매김.

추천 조합 (예: A+B+D+E + I) + 사유 기록.

### 2.6 Step 5. YC 파트너 4명 시뮬 (선택 — 큰 차별화·강함·매우 강함 시)

큰 차별화 결정 또는 카테고리 재정의 시 YC 4인 시뮬:

| 파트너                 | 시뮬 시각                                                                                                                      |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| **Paul Graham**        | 10x on one axis / Live in future / Schlep blindness — _한 축에서 10배 더 나은가, 미래 흐름에 부합하나, schlep을 피하지 않나_ |
| **Garry Tan**          | Why now·you / Schlep moat / Distribution — _지금 왜 이게 가능한가, 너가 왜 만드는가, 해자 있는가_                              |
| **Michael Seibel**     | Founder market fit / Make 10 users love it — _너가 이 사용자를 깊이 아는가, 10명이 사랑할 형태인가_                            |
| **Jessica Livingston** | Distribution path / Retention vs Referral — _사용자가 어떻게 발견하나, 돌아오나·소문 내나_                                     |

각자 _이 차별화로 들어가면_ 평가 1줄. **4명 중 2+명 *go 강함***이면 차별화 충분.

### 2.7 Step 6. 최종 서비스 기획 요약 내용 — 최종 아이디어 + 빌더 가드레일

경쟁 강도를 판정한 직후, 이 단계 마지막으로 **최종 서비스 기획 요약 내용** 4개를 박는다. *남에게 설명할 문서가 아니라, 빌더 자신이 뭘 만들고·안 만들고·검증할지 정하는 최소 가드레일*이다.

**(1) 최종 아이디어** (한 문단):
- **없음 / 약함**: `idea-validation.md` §1.8.1 _초기 확정 아이디어_ 그대로 또는 미세 다듬기 ("없음"이면 §2.4.1 의심 절차 통과가 전제)
- **강함**: 차별화 axis 반영해 _갱신_ — 차별화 지점 명시
- **매우 강함**: §2.8 분기 — 피봇 또는 폐기 결정
- 형식: `[누구]가 [상황]에서 [문제] 겪는 걸 [어떻게] 풀어준다` — [누구]·[상황]이 타겟·맥락을 품는다. 세부 타겟·맥락은 ScreenDesign에서 화면별로 구체화하므로 *여기서 재서술하지 않는다*.

**(2) 비목표 (안 만들 것)** — 형식 `[X] 안 함 — [이유]`. 최소 1개, 그중 **1+개는 프로토타입 스코프**("이번 데모에선 X 안 함 — 이유"). *왜*: 빌드가 싸진 시대엔 안 자르면 화면이 무한 증식 → ScreenDesign·Prototype이 부푼다. 비목표가 그 브레이크.

**(3) 검증 가설** — 형식 `[조건]하면 [예측]할 것이다`. 1~2개, 그중 **1+개는 프로토타입 HTML 데모로 검증 가능한 형태**. Retro 정성 인터뷰 N=5+ 설계의 직접 입력.

**(4) 북극성 (1줄, 선택)** — 2~3년 horizon 방향 한 줄(행동·규모·포지셔닝 중 1개). 이번 검증에 묶인 수치 금지(그건 retro). Prototype 단계의 Claude Design 프롬프트 톤 참조용. 빌더 단계엔 가벼운 방향타로 충분 — 생략 가능.

**코칭**: (2)(3)(4)는 founder 의지라 추정 금지 — AI가 후보 2~3개 제시 → 사용자가 골라 다듬는다(SKILL.md §1.4). 빈칸 압박 대신 후보로.

최종 서비스 기획 요약 내용은 *ScreenDesign·Prototype 진입 입력*으로 사용 (`market-research.md` §최종 서비스 기획 요약 내용에 박음).

**이 요약 = 2단계 시점의 진입 스냅샷** (SKILL.md §2.3) — 아이디어·가설 체인의 시작점이다. 3단계부터 방향이 바뀌어도 **이 문서를 소급 수정하지 않는다** — 새 방향은 바뀐 시점의 단계 산출물 진입 스냅샷에 기술하고, 이 문서엔 필요 시 끝에 포인터 한 줄만 (`→ YYYY-MM-DD 이후 방향은 screen-design.md 진입 스냅샷 참조`).

### 2.8 "매우 강함" 판정 시 — 피봇·폐기 분기

**매우 강함** = 기존 강자(incumbent)가 해자를 구축해서, 차별화 축으로도 이길 가능성이 낮다. **피봇·폐기 권장**.

두 분기:

1. **피봇** — 같은 도메인 _다른 진통제_ 발굴 (인터뷰 노트 다시 봐서 _우리가 놓친_ 시사점 찾기)
2. **폐기** — 다른 아이디어로 새 프로젝트 (`idea-validation.md` 보존, 새 아이디어로 idea-validation-v2 시작)

운영자 모드·동기 frame 적용:

| 모드·동기                  | "매우 강함" 판정 시                          |
| -------------------------- | -------------------------------------------- |
| **풀타임 + 매출**          | 피봇·폐기 권장 강함 (시간·자본 위험 큼)      |
| **사이드 + 매출**          | 피봇 권장 (사이드라 기존 강자 이기기 어려움) |
| **사이드 + 학습**          | "매우 강함"이어도 _학습·dogfooding 가치_ 있으면 진행 OK |
| **부업·취미 + dogfooding** | "매우 강함"이어도 *본인이 매주 쓸 도구*면 진행 OK |

폐기 결정은 강제 X — 사용자 선택. 사유 산출물에 기록.

## 5. 재실행 시 처리

이 스킬은 프로젝트당 1회 — 별도 재실행 시 처리가 없다. 이 단계 결과로 분기:

- **go (ScreenDesign 진입)** — 없음(§2.4.1 의심 절차 통과)/약함, 또는 강함 + 차별화 결정 + 최종 서비스 기획 요약 내용(최종 아이디어·비목표·가설) 후
- **no-go (피봇)** — 매우 강함 + 피봇 선택. 같은 도메인 새 진통제로 `idea-validation.md` 갱신 후 재진입
- **no-go (폐기)** — 매우 강함 + 폐기 선택. 새 프로젝트에서 처음부터

**뒤 단계 진행 중 피봇이 나면** — 이 문서로 되돌아와 본문·요약을 재정합하지 않는다. 새 방향은 영향받는 단계(대개 3단계)의 진입 스냅샷에 기술 (SKILL.md §2.3). 이 문서는 frozen 기록으로 보존, 끝에 포인터 한 줄만 추가.

## 6. 좋은 예 vs 나쁜 예

예시는 도메인 무관 — *유명 서비스*로 패턴만 본다. 핵심은 **구체·검증 가능·간결**(좋은) vs **추상·일반론**(나쁜).

### 6.1 경쟁자 발견

**좋은 예** (0번 경쟁자 + Agent deep research + 약점 구체):

> 할 일 앱 — 0번: "메모 앱 + 머릿속" (인터뷰 4명 중 3명). 경쟁자 N=5: Todoist · TickTick · Things · Notion · Apple 미리알림. 약점: "Notion = 무겁고 느림 / Apple 미리알림 = 협업 X".

**나쁜 예** ← 사용자 1명이 말한 것만:

> 경쟁자: Notion.

### 6.2 강도 판정

**좋은 예** (점유·해자 근거):

> 매우 강함. 범용 검색엔진 진입 — Google 점유 90%+. 해자 체크: 네트워크 효과 O(검색량→품질 개선 고리) · 전환 비용 X · 데이터 축적 O(수십 년) · 브랜드 O · 규제 X. 정면 승산 낮음.

**나쁜 예** ← 추측:

> 없음. 별로 없어 보임.

### 6.3 차별화 축(axis)

**좋은 예** (포화 시장 + 단일 축 10배):

> Superhuman: 이메일(포화 시장)에 A(좁은 타겟: 파워유저) + E(속도 단일 축 10배). "Gmail보다 빠른 이메일" 한 줄로 선다.

**나쁜 예** ← 막연:

> UX를 더 좋게 만들어 이긴다.

### 6.4 카테고리 재정의

**좋은 예** (흐름 자체를 바꿔 직접 경쟁 회피 — YC 단골 사례):

> Airbnb. 기존 카테고리 "호텔을 예약한다" → 재정의 "남의 집 빈방에 묵는다". 호텔 체인과 같은 링에서 안 싸운다.

**나쁜 예** ← 같은 카테고리 안 표면 차별:

> 더 싸고 예쁜 호텔 예약 앱.

### 6.5 모드·동기별 적용 차이

**나쁜 예** (모드 무시):

> 사이드 + 매출인데 "매우 강함"에 "그래도 도전!" → 자본·시간 부족으로 흐지부지.

**좋은 예** (모드 맞춰 스코프 조절):

> 사이드 + 매출 & 강함 → 축 1~2개로 좁혀 *작은 쐐기(wedge)*로 진입(1년 운영 가능). / 풀타임 + 매출 & 강함 → 축 3+ 묶어 정면 도전.

### 6.6 비목표 (최종 서비스 기획 요약 내용)

**좋은 예** (사업·프로토타입 분리, 이유 포함 — Airbnb 초기):

> - 결제·보험 자체 구축 안 함 — 외부로 충분, 핵심은 매칭 (사업 차원)
> - 이번 데모엔 호스트용 화면 안 만듦 — 게스트 예약 흐름 먼저 검증 (프로토타입 스코프)

**나쁜 예** ← 차원·이유 없음:

> - 기능 너무 많이 안 만듦

### 6.7 검증 가설 (최종 서비스 기획 요약 내용)

**좋은 예** (조건→측정, 분리 — Airbnb):

> - 게스트가 사진·후기를 보면 5분 내 예약을 결정한다. (사업 — MvpLaunch 검증)
> - 데모로 검색→상세→예약 흐름 보여주면 5명 중 3+명이 "직접 올려볼게"라 한다. (프로토타입 — retro 입력)

**나쁜 예** ← 검증 불가:

> - 사용자가 좋아할 것.

## 7. 사용자 응대 톤 + 인터뷰 코칭

톤·코칭은 SKILL.md §1.3·§1.4 따름. 핵심 동작:

- §2 순서 따름: 발견 → 분석 → 강도 → axis → 시뮬 → 갱신. Step 건너뛰기 금지.
- **운영자 모드·동기 frame** (1-idea-validation §1.3) 명시 — 사이드 모드면 axis 추천 시 _작은 조각_ 우선, 풀타임이면 정면 도전 가능.
- **사용자가 *경쟁자 없다*고 우길 때** — Agent로 한 번 더 검증 권장 (자기 검열). 진짜 없을 수도 있지만 못 찾은 것일 가능성 명시. 그래도 없으면 §2.4.1 의심 절차.
- **사용자가 모든 axis 욕심**낼 때 — 1~2개로 좁히기 권장 (사이드 모드 시간 보호). 욕심 안 자르면 이 단계가 안 끝난다.
- **"매우 강함" 판정에 *폐기 결정***은 강제 X — 사용자 선택. 모드·동기 frame 명시.
- **정리한 산출물 보여주고 확인** — "이거 맞아? 고칠 데 있어?"
- 모르는 부분은 `추후 결정: ...`으로 명시.
