이 단계는 **프로토타입을 진짜 서비스로 만드는** 단계다. LLM 시대에 이 수준 빌드는 싸다 — 그래서 병목은 빌드가 아니라 (1) **범위 절제**(명세 밖 기능이 슬금슬금 붙는 것)와 (2) **계측 정확성**(7단계 MvpLaunch 판정이 여기서 심은 이벤트로 이뤄짐)이다. 이 references는 그 둘을 지키는 가드에 집중하고, 빌드 자체는 초안 모드(SKILL.md §1.5)로 완주한다.

**문서를 새로 쓰지 않는다** — `screen-design.md`(무엇을·어떤 화면으로) + `prototype-ver-{N}/` README(어떻게 생겼는지 설계도) + reference HTML(픽셀 정답지)이 이미 스펙이다. PRD·Architecture 문서 작성 금지. 이 단계가 새로 만드는 문서는 `mvp-build.md`(스택·계측·배포 기록) 하나다.

## 1. 단계 목표

- **배포된 실서비스** — 프로젝트 루트 폴더에 앱 코드(웹사이트·모바일 앱·데스크톱 등 형태 무관), 실백엔드·DB·auth 연동, 공개 URL로 접속 가능.
- **퍼널 계측 심기** — 유입·가입·핵심행동·재방문·결제 5종 이벤트가 실제로 찍힘. 7단계(MvpLaunch) 판정의 데이터 원천.
- **(지불 가설 있으면) 실결제** — market-research 검증 가설에 지불 검증이 포함되면 실결제 연동 (fake door 아님).
- **`mvp-build.md`** — 스택 결정(ADR 링크)·계측 이벤트 정의·배포 정보 기록.

**SoT 책임** (SKILL.md §2.1): 스택 결정 · 퍼널 계측 이벤트 정의 · 배포 정보 = `mvp-build.md`. 화면·플로우 명세는 screen-design SoT 참조만 — 여기 중복 금지.

## 2. 진행 절차

### 2.1 입력 확인

- `prototype-ver-{N}/` (가장 큰 N) — README 설계도 + reference HTML 정답지 + `src/tokens/`·컴포넌트. **빌드의 메인 입력.**
- `proto-retro.md` §MvpBuild 입력 — 인터뷰에서 호응/거부 받은 부분, 타겟 narrow, 비목표 확장.
- `market-research.md` §최종 서비스 기획 요약 내용 — 비목표(빌드 범위 가드레일) + 검증 가설(지불 검증 포함 여부 → 실결제 필요 판단).
- 옛 단계 evidence는 SKILL.md §2.3 검토 절차(4 체크 질문 + 라벨) 거쳐 사용.

### 2.2 스택 결정 — ADR (인터뷰 모드)

스택 선택은 **되돌리기 비싼 갈림길**이라 에스컬레이션 대상(SKILL.md §1.5) — AI가 후보 2~3개 + 추천 + 이유를 표로 제시하고 사용자가 결정한다. 결정 기준:

- **최단 경로 우선** — 붙이면 바로 되는 관리형 서비스 (호스팅+DB+Auth 통합형). MVP는 검증용이지 자산용이 아니다.
- **prototype 토큰·컴포넌트 이식이 자연스러운 프레임워크** — zip의 `src/`가 React 계열이므로 React 계열이 기본 후보.
- **결제**: 지불 가설 있으면 실결제 PG(토스페이먼츠·Stripe 등) — 심사 소요 기간을 미리 확인 (수 일 걸릴 수 있어 빌드와 병렬 진행).

결정은 `planning/adr.md`에 기록 (SKILL.md §3.1 — DB·Auth·Hosting·결제 전부 ADR 트리거).

### 2.3 빌드 — 초안 모드 완주

AI가 **배포 가능한 상태까지 완주**한다 (SKILL.md §1.5). 중간에 화면 하나 만들 때마다 확인받지 않는다 — 리뷰는 배포 URL 클릭으로 한 번에.

**범위 가드 (강제)**:

- `screen-design.md` 명세에 없는 화면·기능 추가 금지. 빌드 중 "이것도 있으면 좋겠는데" 신호는 `추후 결정:`으로 mvp-build.md에 마킹만.
- market-research §비목표 화면·기능은 빌드에도 없음.
- prototype에서 MVP 마스크 처리했던 영역(3-screen-design §6.6.A) — 이번에 *실제로 작동*시켜야 하는지 proto-retro 입력으로 판단. 가설 검증에 불필요하면 마스크 유지.
- 다크모드·다국어·반응형 정교화·성능 튜닝·CI 정교화는 범위 밖 (§5).

**보안 baseline**: auth·API·업로드 코드를 만지면 `.claude/rules/threat-model.md`가 자동 로드된다 — 그 baseline(http-only cookie·권한 검사·IDOR 회피·rate limit·에러에 stack 미노출 등) 준수. 광고로 모르는 사람들이 실제로 들어오는 코드다.

**에스컬레이션**: 명세와 모순 발견·스키마가 크게 갈리는 선택·외부 서비스 추가 필요 시 멈추고 배치 질문. 그 외 애매함은 합리적 가정 + `추후 결정:` 마킹 후 계속.

### 2.4 퍼널 계측 심기 (이 단계의 핵심)

7단계(MvpLaunch) 판정은 **여기서 심은 이벤트가 전부다** — 계측이 빠지면 광고비를 쓰고도 판정 불능. 5종 기본 이벤트:

| # | 이벤트 | 정의 (예시 — 서비스에 맞게 구체화) | 퍼널 위치 |
|---|---|---|---|
| 1 | 유입 | 랜딩 도착 (유입 채널 구분 파라미터 포함 — 발품 vs 광고 vs 기타) | 최상단 |
| 2 | 가입 | 계정 생성 완료 | 전환 1 |
| 3 | 핵심행동 | *이 서비스의 aha 순간* 1회 완료 (예: 매물 진단 1회) — 무엇으로 할지 사용자와 결정 | 활성화 |
| 4 | 재방문 | 가입 후 7일 내 재방문 | retention |
| 5 | 결제 | (지불 가설 있으면) 결제 완료 / (없으면) 가격 페이지 도달 등 지불 의사 프록시 | 수익 |

- **채널 구분이 필수** — 발품 유입과 광고 유입을 구분 못 하면 7단계(MvpLaunch) 이중 채널 판정이 무너진다. UTM 파라미터 등으로 구분.
- 도구는 스택 결정(§2.2)에서 함께 — 가벼운 것 우선 (관리형 analytics 또는 자체 이벤트 테이블).
- **이벤트 정의를 `mvp-build.md`에 표로 기록** — 7단계(MvpLaunch) launch-plan이 이 정의를 그대로 참조한다 (이름·정의 drift 방지).

### 2.5 배포 + 스모크 테스트

1. 배포 (관리형 호스팅 기본). main 직접 push 금지 — PR 워크플로 (repo 훅이 막음).
2. **사용자가 배포 URL을 브라우저로 직접 클릭** — 핵심 시나리오 끝까지 통과 + 가입·핵심행동 이벤트가 실제로 찍히는지 확인 (CLAUDE.md 검증 룰: AI가 자동 도구로 검증·사인오프하지 않음).
3. 실결제 연동했으면 **실결제 테스트 1건** (결제 → 환불) — 사용자가 직접.
4. 발견된 문제는 초안 모드로 수정 → 재배포 → 재확인 (자율 재작업 상한 2회 — SKILL.md §1.5).

## 3. 완료 체크리스트

다음이 모두 충족되어야 다음 단계(MvpLaunch)로 진입.

- [ ] **공개 URL 접속 가능** — 배포 완료, 핵심 시나리오가 실데이터로 끝까지 돈다.
- [ ] **screen-design 명세 커버** — 명세 화면 누락 0 (비목표·마스크 유지 영역은 제외 명시).
- [ ] **퍼널 이벤트 5종이 실제로 찍힘** — 사용자가 직접 클릭해 이벤트 발화 확인. 채널 구분 동작.
- [ ] **(지불 가설 있으면) 실결제 1건 테스트 완료** (결제 → 환불).
- [ ] **`mvp-build.md` 작성됨** — 스택(ADR 링크)·계측 이벤트 정의 표·배포 정보.
- [ ] **(사용자 검증) 사람이 브라우저로 배포물 클릭 검토 완료.**

체크리스트 미충족 시 SKILL.md §4 단계 진입 평가 정책에 따라 사용자에게 선택지 제시.

## 4. 산출물 스펙

- **앱 코드**: 프로젝트 루트 폴더 (`package.json`·`src/` 등 — 스택 표준 구조 그대로).
- **`planning/mvp/mvp-build.md`**:

```markdown

created_at: YYYY-MM-DD
updated_at: YYYY-MM-DD

# 스택

# 퍼널 계측 이벤트 정의
| # | 이벤트 | 정의 (이 서비스 기준 구체) | 구현 위치 |
|---|---|---|---|
| 1 | 유입 | ... | ... |
| ... | | | |

# 배포 정보
- URL: ...
- 배포 방식: (예: main merge → 자동 deploy)

# 추후 결정
- [빌드 중 마킹한 항목들]

# 빌드 중 발견 (7단계 MvpLaunch·production 입력)
- [명세 모순·개선 아이디어 — screen-design을 역방향 수정하지 않고 여기 기록]
```

## 5. 여기서 안 하는 것 (의도적)

- ❌ PRD·Architecture·Design 문서 작성 — screen-design + zip README가 스펙. 문서 ceremony 금지.
- ❌ 명세 밖 기능·화면 추가 — `추후 결정:` 마킹만.
- ❌ 다크모드·다국어·반응형 정교화·성능 튜닝·부하 대비 — production 영역.
- ❌ CI·테스트 스위트 정교화 — 검증 가설과 무관한 인프라 투자는 production에서.
- ❌ 광고·채널 준비 — 7단계(MvpLaunch) 영역 (단 결제 PG 심사처럼 리드타임 긴 것만 병렬 시작 OK).

## 6. 사용자 응대 톤 + 코칭

- **톤**: SKILL.md §1.3대로 반말·친근·짧게.
- **진입 안내**: "프로토타입 반응 봤으니 이제 진짜 서비스로 만들 차례야 — 백엔드·DB 붙이고 배포까지. 스택 몇 개만 정하면(내가 후보+추천 낼게) 빌드는 내가 완주해서 배포 링크로 보여줄게. 너는 링크 열어서 클릭으로 확인하면 돼."
- **스택 결정**: 후보 2~3개 표 (옵션/장단점/추천/이유) + 배치 질문. 결정 후 ADR 기록 확인.
- **빌드 중**: 잘게 확인 요청 금지 (초안 모드). 에스컬레이션 기준 해당 시에만 멈추고 배치로 질문.
- **완료 시**: 배포 URL + "핵심 시나리오대로 클릭해봐 + 이벤트 찍히는 것도 같이 확인하자" + 체크리스트 현황.
- **범위 압박 대응**: 사용자가 "이 기능도 넣자" 하면 — screen-design 명세·비목표와 대조해 짚고, 가설 검증에 필요한지 확인. 필요 없으면 "런치 검증(7단계) 끝나고 production에서" 권유 (강제는 아님 — 사용자가 원하면 명세 먼저 갱신 후 빌드).
