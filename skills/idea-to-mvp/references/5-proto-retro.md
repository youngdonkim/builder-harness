ProtoRetro는 **통과 기준③** — 프로토타입 구간(1~4단계)의 종착점이자 **MvpBuild(6단계) 진입 판정**이다. market-research 최종 서비스 기획 요약 내용의 검증 가설이 정성 인터뷰로 어떤 신호를 받았는지, 프로토타입에서 어떤 화면·문구가 호응을 얻었는지, AI 도구 작업에서 잘됐고 아쉬운 점은 무엇인지를 정리. 이 회고가 6단계 MvpBuild(빌드 범위)·7단계 MvpLaunch(런치 기준 후보)의 입력으로 들어간다. 목적은 **치명 결함 거르기** — 최종 판정은 통과 기준④(MvpLaunch — 실제 사용자 행동)가 한다.

## 1. 단계 목표

- **정성 인터뷰 결과 정리**: N=5+ 타겟 사용자 인터뷰 **권장** — 행동 의향·페이지 인상·문구 호응. (직장+사이드 빌더에게 모집은 현실적으로 어렵다 — 미달로도 판정 가능하되 근거 약함을 명시. 최종 검증은 어차피 통과 기준④ 실제 사용자 행동이 한다.)
- **핵심 가설 검증 결과**: 최신 '이번 결론'의 검증 가설(SKILL.md §2.2) → 검증됨 / 일부 / 미검증.
- **잘된 점·아쉬운 점**: screen-design·prototype 작업·인터뷰 진행.
- **발견 사항 룰화 판정**: screen-design 작업·prototype 제작 중 발견된 패턴 — 룰화 / spot fix / 코멘트.
- **Painkiller test 평가 평가**: 3항목(문제 강도·타겟 specificity·Pull 신호)으로 go/no-go 판정 — 디테일은 §2.5.1.
- **go/no-go 판정 · 다음 분기**: Painkiller 평가 + 운영자 직관 → 실서비스 빌드(MvpBuild)로 갈지 + 3분기 결정 (§2.6).
- **6·7단계 입력**: MvpBuild 빌드 범위·MvpLaunch 런치 기준 후보에 가져갈 인사이트·새 가설.
- **워크플로우 개선 제안** (선택): idea-to-mvp 스킬 피드백.

### 1.1 왜 prototype 단계엔 Painkiller test인가 (사용자에게 설명)

retro 진행 시 사용자에게 **왜 이런 기준을 적용하는지** 첫 안내에 짧게 설명한다.

설명 골자 (사용자에게 그대로 또는 paraphrase해서 전달):

> Prototype은 실코드·실배포 전 단계라 retention·NPS·매출 같은 정량 지표가 안 나와. 이 단계에서 측정해야 할 건 **"우리가 풀려는 문제가 진짜 painkiller(진통제)인가, vitamin(있으면 좋고 없어도 사는 것)인가"** 하나야.
>
> 진통제 신호는 정성 인터뷰 N=5+에서 나와 — 그래서 YC 학파에서 "Make something a *few* people love"라고 강조해. 100명이 미지근한 것보다 5명이 펄쩍 뛰는 게 훨씬 의미 있음.
>
> Prototype은 **두 평가**로 painkiller 검증:
> - **진입 평가 (Stage 1 IdeaValidation, 이미 통과한 상태)**: 항목1(임시방편 보유)·항목2(타겟 specificity) — *데모 없이 인터뷰만으로* 검증. prototype 빌드 진입 자격 확인용.
> - **종료 평가 (지금 이 retro 단계)**: 항목1 *full* 재검증 (N=5+ 인터뷰로) + 항목3(Pull 신호 — 데모 후 자발 베타 의향) *처음 측정*. MvpBuild 진입 자격 결정.
>
> 왜 entry/exit 두 단계로 나눠? — 항목1은 데모 없이도 인터뷰만으로 측정 가능해서 *비싼 화면 빌드 전*에 거를 수 있음. 항목3은 데모가 있어야 측정 가능해서 retro에서 처음.
>
> 최종 판정(통과 기준④ MvpLaunch)에선 기준이 달라져 — 거기선 실제 사용자 행동 퍼널(가입·활성화·재방문·결제) 지표를 본다. Painkiller test는 이 통과 기준③ 전용.

## 2. 진행 절차

### 2.1 단계 시작 — 이전 정리 내용을 보여주고, 이번 단계가 할 일을 안내한다

이 단계에 들어오면 가장 먼저 사용자와 "지금까지 어디까지 왔고, 이제 무엇을 할지"를 맞춘다. 순서대로:

1. **이전 단계까지 정리된 내용을 먼저 보여준다.** 프로토타입에 입력됐던 방향(대개 `screen-design.md` 머리)과, 프로젝트 루트의 `[projectName]-proto.html`(테스트 대상 프로토타입)이 준비돼 있음을 짧게 요약해 보여준다 — 단, 프로토타입을 만들며 그 방향이 통째로 바뀌었을 수 있음을 전제로(SKILL.md §2.2 대전제).
2. **그 내용을 기반으로 이번 단계가 무엇을 하는지 설명한다.** 프로토타입을 사람들에게 보여준 인터뷰 결과를 정리해서, 검증 가설이 어떤 신호를 받았는지 확인하고 실서비스 빌드(MvpBuild)로 갈지 판정하는 단계(통과 기준③)라고 안내한다. §1.1의 "왜 Painkiller test인가" 설명도 이때 함께.
3. **`proto-retro.md` 머리에 `# 이전 결론 → 이번 결론` 섹션을 *두 패스*로 적는다** (SKILL.md §2.2):
   - **이전 결론 — 지금(진입 시).** 프로토타입 단계의 결론은 곧 **최종 `[projectName]-proto.html`(코드)**이다. 그러니 이전 결론 4개(최종 아이디어·비목표·검증 가설·북극성)를 **이 HTML에서 읽어 뽑는다.** HTML엔 4개가 문장으로 안 적혀 있으니 AI가 화면·카피·기능·데이터에서 **추론 → 사용자에게 보여주고 확인**받아 확정한다(§1.4). **DEV-NOTES·README·옛 피봇 메모 말고 오직 최종 HTML만 근거.** screen-design 입력과 달라졌으면(피봇) 그 바뀐 상태가 그대로 이전 결론이 된다.
   - **이번 결론 — 나중(인터뷰 후, §2.5).** 이 프로토타입으로 인터뷰한 *결과*로 정리한다. 진입 시엔 `추후 결정 — 인터뷰 후`로 비워두고, §2.5 최종화에서 채운다. 인터뷰는 미세조정만이 아니라 아이디어·가설을 **확 뒤집을 수도** 있다.
   - **무엇이 왜 바뀌었나** = 이전 결론(HTML) 대비 이번 결론(인터뷰 후)의 델타. 인터뷰가 아무것도 안 바꿨으면 이번 결론 = `변경 없음 — 이전 결론 그대로` 한 줄.
4. **이미 `proto-retro.md`가 있으면** 읽고 §3 체크리스트로 검증해 부족한 부분만 보완한다.

**회고 범위**: ProtoRetro는 매번 그 아이디어의 첫 회고다 — 정성 인터뷰 결과·go/no-go 시그널 위주로 짧게. 이 회고 자체가 6·7단계(MvpBuild·MvpLaunch)의 입력이 된다. (다른 아이디어로 새 프로젝트를 열면 직전 아이디어를 물려받지 않으므로 "재방문" 케이스가 없다.)

### 2.2 자동 데이터 수집 (AI 단독)

AI가 prototype 산출물·인터뷰 노트를 읽고 데이터 추출. 인터뷰 없이.

**소스**:
- 프로젝트 루트 `[projectName]-proto.html` → **이전 결론의 유일 소스.** 이 프로토타입이 실제 구현한 4개(최종 아이디어·비목표·검증 가설·북극성)를 화면·카피·기능·데이터에서 추론(→ 사용자 확인). **DEV-NOTES·README 등 부차 문서는 결론 도출에 안 씀 — 최종 HTML만 SoT** (SKILL.md §2.2).
- `screen-design.md` → 프로토타입에 *입력됐던* 방향 (맥락용 — 프로토타입이 여기서 피봇했는지 대조). 이전 결론의 직접 소스 아님.
- `idea-validation.md` → Stage 1 인터뷰 노트·진입 Painkiller 평가 결과 (항목1·2 base)
- `interview-notes.md` (별도 파일 또는 proto-retro.md 인라인) → N=5+ 사용자 인터뷰 응답·관찰 → **이번 결론**·가설 검증
- `git log -- [projectName]-proto.html` → 프로토타입 iterate 횟수(덮어쓴 커밋 수)·작업 기간

**자동 추출 결과 예**:

```
prototype 통계 (자동 수집):
- prototype ver: 11회 iterate
- 작업 기간: 2026-04-15 ~ 2026-05-23 (38일)
- 최종 화면 수: 8 (landing·step1~4·generating·result·product page)
- 인터뷰: 5명
- git commits: 47개
```

### 2.3 proto-retro.md 초안 작성 (AI 단독)

자동 데이터로 proto-retro.md 초안을 **직접 작성**. 사용자 인터뷰는 초안 review 단계로 미룸 — 추상 질문에 답하는 것보다 **구체 초안 review가 효율·정확도 ↑**.

**원칙**:
- 자동 추출되는 모든 항목은 초안에 채움
- 자동 추출 안 되는 항목은 **`추후 결정: <무엇을 확인할지>` 마커**로 명시. **추측으로 채우지 말기.**
- frontmatter `status: draft` — 사용자 확정 전까지

**자주 추후 결정가 되는 카테고리** (자동 추출 거의 안 됨):
- 정성 인터뷰 응답 (사용자가 interview-notes.md 별도 제공 필요)
- 핵심 가설 검증 결과 (인터뷰 결과 종합 필요)
- 새로 발견된 가설 (사용자 주관)
- 잘된 점·아쉬운 점 중 dynamics·취향 영역
- go/no-go 판정·다음 분기 (사용자 결정 — §2.6)

**자동 추출되는 카테고리** (추후 결정 안 함, 추출 못 하면 "데이터 없음" 명시):
- prototype iterate 횟수·화면 수 (`[projectName]-proto.html`)
- git commits·작업 기간 (`git log`)
- 산출물·rules·hooks·skills 변화 추적

### 2.3.1 발견 사항 — 룰화 판정 3분류

screen-design 작업·prototype 제작 중 발견된 패턴·이슈를 다음 3분류로 판정. **룰화는 비용**(LLM context 소비·트리거 매핑 약화) — 가치 통과한 것만 룰화.

| 분류 | 기준 | 처리 |
|---|---|---|
| **룰화** | (a) **재발 위험**: 다음 프로젝트/유사 작업에서 같은 함정 가능성 + (b) **횡단 가치**: 이 프로젝트뿐 아니라 유사 프로젝트에도 적용 가능 | `.claude/rules/<topic>.md` 신규 또는 기존 룰 보강. paths frontmatter로 트리거 매핑. CLAUDE.md 컨벤션 한 줄 추가 검토 |
| **spot fix** | 일회성 수정. 같은 패턴 재발 가능성 낮음 | 코드만 수정. 룰화 X |
| **코멘트** | 사용자·팀 dynamics·취향·일회성 판단 | 어디에도 박지 않음. proto-retro.md "잘된 점·아쉬운 점"에만 |

**판정은 둘 다 통과해야 룰화** — 재발 위험만 있고 횡단 가치 없으면 spot fix. 횡단 가치만 있고 재발 위험 낮으면 코멘트로.

prototype은 본격 빌드 없어 룰화 항목 적을 수 있음 — prototype 디자인 시스템·UX 카피 패턴 위주.

발견 사항 표 예시:

```markdown
## 발견 사항 — 룰화 판정

| # | 내용 | 분류 | 처리 |
|---|---|---|---|
| 1 | prototype hero 카피가 over-promise로 박힘 (ux-writing-reviewer 기준 위배) | 룰화 | `ux-writing-reviewer` 에이전트에 "over-promise 회피" 보강 |
| 2 | prototype ver 늘어날수록 ver-N 파일 누적, 어느 게 SoT인지 헷갈림 | 코멘트 | retro 아쉬운 점 |
| 3 | product.html 마커가 소비자 view엔 노이즈 | spot fix | `?marker=on` param 분기 |
```

### 2.4 추후 결정 채우기 (선택적 인터뷰)

§2.3 초안을 사용자에게 보여주고 **추후 결정 항목만** 답 받음. 고정된 질문 목록이 아니라 **초안 추후 결정 개수만큼** N개.

```
proto-retro.md 초안 끝났어. 추후 결정 [N]개 답해줘:

1. {추후 결정 항목 1} → {무엇을 확인할지}
2. {추후 결정 항목 2} → ...
...

답 한 번에 줘도 되고 하나씩 줘도 OK.
```

답이 모호하면 SKILL.md §1.4 코칭으로 한 번 더 구체화 요청. 사용자가 "모르겠어"면 후보 2~3개 제시. 추후 결정 답이 부담스러우면 "보류 — 사유" 명시 허용.

### 2.5 proto-retro.md 최종화

추후 결정 답을 반영해 최종화:
- 각 추후 결정 마커 → 사용자 답으로 대체 (또는 "보류 — 사유" 명시)
- **`# 이전 결론 → 이번 결론`의 '이번 결론' 확정 (2패스 두 번째)** — 진입 시 `추후 결정 — 인터뷰 후`로 비워뒀던 이번 결론을, 인터뷰 결과로 정리해 채운다. 인터뷰가 아이디어·비목표·검증 가설을 바꿨으면 4개를 새로 기술 + `## 무엇이 왜 바뀌었나`(이전 결론 대비), 안 바꿨으면 `변경 없음 — 이전 결론 그대로` 한 줄. **이 '이번 결론'이 곧 MvpBuild의 '이전 결론'으로 넘어간다** — 인터뷰까지 거친 최신 결론이 SoT.
- frontmatter `status: draft` → `status: confirmed`
- 사용자에게 최종 확인

### 2.5.1 종료 Painkiller 평가 (Entry 결과 + Retro 신규 측정)

proto-retro.md 추후 결정 채우기(§2.4)에서 사용자가 인터뷰 결과를 답하면, AI가 그 답을 **Painkiller 3항목**으로 평가. 진입 평가(Stage 1)는 이미 통과한 상태이므로 retro에선 다음 두 가지 작업:

1. **항목1·2 재검증** — Stage 1에서 작은 표본으로 검증된 신호가 N=5+ full 인터뷰에서도 유지되나? (강화 / 유지 / 약화 / 무너짐)
2. **항목3 처음 측정** — 데모 보여준 후 자발적 베타 의향. 이건 데모 있어야 측정 가능했으니 retro에서 처음.

**3항목 기준** (§1.1 골자 그대로):

> **SoT 주의**: 항목1·2의 *통과 기준 정의*는 stage-1(`1-idea-validation.md` §2.7.2 강도표·§2.8 등급표)이 원천이다. retro는 그 기준으로 N=5+ 재검증만 하고 *기준을 다시 정의하지 않는다* — 정의 drift 방지. 항목3(Pull)만 retro 신규.

| # | 항목 | 통과 기준 | retro에서 작업 |
|---|---|---|---|
| 1 | **문제 강도** (Painkiller) | stage-1 §2.8 1항목 등급 그대로 (★★★+★★ 절반↑·★★★ 1명↑; 강도 분류는 stage-1 §2.7.2 강도표) | **Entry 결과 재검증** — Stage 1 N=3~5 신호가 retro N=5+에서도 유지되나. 새 인터뷰이의 hack 증거를 §2.7.2 강도표로 분류·카운트, Stage 1 결과와 비교 |
| 2 | **타겟 specificity** | stage-1 §2.8 2항목 등급 그대로 (10명+ 실명 list-up) | **Entry 결과 confirmation** — Stage 1에서 list-up 가능했던 게 retro 시점에도 유효한가. 보통 큰 변화 없음, 변화 있으면 사유 기록 |
| 3 | **Pull 신호** (retro 신규) | 데모 본 후 2명+ 자발 "써보고 싶다" 요청 | **처음 측정** — Stage 1엔 데모 없어 측정 불가했음. 인터뷰 노트에서 자발 의향 카운트 |

**평가 결과 표기** (proto-retro.md에 박힘):

```markdown
# Painkiller 평가 (종료 — Stage 1 결과 + Retro 신규)

| 항목 | 통과 기준 | Stage 1 결과 | Retro 결과 | 근거 |
|---|---|---|---|---|
| 1. 문제 강도 | N=5+ 중 3명+ hack 보유 | ✅ / ⚠️ / ❌ (idea-validation.md) | ✅ 강화 / ✅ 유지 / ⚠️ 약화 / ❌ 무너짐 | "Stage 1 N=4 중 3명 → retro N=6 중 4명, 동일 패턴" |
| 2. 타겟 specificity | 10명+ 실명 list-up | ✅ / ⚠️ / ❌ | ✅ 유효 / ⚠️ 변화 / ❌ 무너짐 | "Stage 1 list-up 10명 유효 / segment 좁힘으로 8명으로 줄어 추가 필요" |
| 3. Pull 신호 | 2명+ 자발 베타 의향 | N/A (데모 전 측정 불가) | ✅ / ⚠️ / ❌ | "사용자 A·C가 데모 후 자발 요청" |

**평가 종합 판정**: [항목1 ✅ + 항목3 ✅ = go 강함 / 항목1 약화 또는 항목3 ❌ = no-go 위험 / 항목1 무너짐 = no-go 강함]
```

**Entry → Exit 변화 해석**:
- **항목1 강화/유지** + **항목3 ✅** → painkiller 검증 완전. MvpBuild 진입 강함.
- **항목1 약화** → Stage 1 표본이 작아 흔들렸을 수도. 종합 직관 + 다른 항목 보고 결정.
- **항목1 무너짐** → painkiller가 진짜 아니었음. no-go 거의 확실. 학습으로 보존.
- **항목1 ✅** + **항목3 ❌** → 문제는 painkiller지만 *우리 솔루션*이 pull 못 만듦. 솔루션 방향 재검토 (피봇 아이디어).

**3항목 모두 통과해야 go인가?** 아니. 평가는 *근거 제공용* 도구. 운영자 직관 + 평가 종합으로 판정. proto-retro.md에 **어느 항목이 약한지, 왜 그래도 go인지** 명시.

**왜 binary 아닌 ✅/⚠️/❌ 3단계인가**: prototype은 작은 표본이라 binary 통과·실패가 흔들리기 쉬움. "통과는 아닌데 신호 있음(⚠️)"을 분리해 직관 판정 근거 풍부.

### 2.6 다음 분기 결정 + 종료 처리

proto-retro.md 초안의 go/no-go 판정이 채워지면, 사용자에게 **3분기를 안내해 다음을 정하게 한다.** 결정은 proto-retro.md "# go/no-go 판정 · 다음 분기"에 기록.

```
프로토타입 회고 끝났어. 이 아이디어 진짜 서비스로 키울지 정하자:

- 실서비스 빌드 (go): 가망 보임 → 같은 스킬의 6단계(백엔드·DB 붙여 배포)로 계속. 이 회고의 빌드 범위·런치 기준 후보를 base로.

- 종료/보류: 지금 결정 안 함.

어느 쪽?
```

프로토타입 구간은 실코드·실배포 전이라 PSF·wow·WTP를 정식 판정하지 않는다 — 그래서 **YC 학파의 Painkiller test**(§2.5.1)를 평가로 쓴다: "문제 강도(임시방편 보유)·타겟 specificity·Pull 신호" 3항목 정성 평가가 go/no-go 근거. 평가 통과 + 운영자 직관 종합으로 판정한다. "솔루션을 개선해야 한다"는 결론이면 그건 go(MvpBuild 진입)다 — 솔루션 개선이 곧 6·7단계가 하는 일이라 별도 분기가 아니다.

**공통 종료 처리** (분기 무관):
- proto-retro.md frontmatter `status: draft` → `confirmed`
- git tag (선택): `git tag proto-gate && git push --tags`

**분기별 처리**:

| 분기 | 다음 액션 |
|---|---|
| **MvpBuild 진입 (go)** | `/clear` 후 `/idea-to-mvp` 재호출 안내 (SKILL.md §3.2) — SKILL.md §1.2가 6단계를 자동 판별해 이어간다. 복사·폴더 생성 없음 (같은 산출물 폴더 계속). |

사용자가 지금 바로 다음 단계를 안 가도 됨 — 분기를 proto-retro.md에 기록해두면, MvpBuild 진입은 나중에 스킬 재호출 시 이 회고를 읽고 이어받는다.

## 5. 좋은 예 vs 나쁜 예

- **정성 인터뷰 결과** — 좋은: "N=5명 중 3명 베타 의향. 호응: '녹음만 넣으면 끝' 흐름. 거부: '자동 요약이 결정사항을 다 잡아줄까?' 의심 (인터뷰이 A, C)". / 나쁜: "다들 좋아함".
- **가설 검증** — 좋은: "가설1 일부 검증, 5명 중 3명 의향, 1명 보류(사유: 녹음 보안 걱정), 1명 거부(사유: 이미 팀 막내가 정리해줌)". / 나쁜: "가설1 검증됨".
- **6·7단계(MvpBuild·MvpLaunch) 입력** — 좋은: "Must 기능에 '정리본 공유' 유지 + 런치 기준 후보에 '가입자 [Y]%가 1주 내 재방문' 추가, 사유: 인터뷰 5명 중 4명이 '정리한 다음 팀이 실제로 보는지 궁금' 답함". / 나쁜: "retention 추가".
- **추후 결정 표기** (§2.3 초안 시점) — 좋은: "정성 인터뷰 응답: 추후 결정 — interview-notes.md 작성 또는 인터뷰 결과 직접 입력 필요". / 나쁜: "인터뷰 결과: 다들 좋아함" (자동 추출 못 한 걸 추측으로 채움).
