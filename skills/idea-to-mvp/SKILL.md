이 스킬은 한 번 호출되면 7단계 중 **현재 단계**를 판별하고 해당 단계 references를 읽어 진행한다. 가령 처음이라면 1단계(IdeaValidation)부터 순서대로 실행하고, 만약 2단계(MarketResearch)까지 했다면 3단계(ScreenDesign)부터 순서대로 실행한다.

## 1. 진입 흐름

### 1.1 스킬 범위 — 아이디어에서 MVP까지, 프로젝트당 1회

이 스킬은 아이디어 하나를 MVP 검증까지 끌고 가는 7단계 과정이다. **한 프로젝트(저장소)는 아이디어 하나만 다룬다** — 진행 중 아이디어를 다듬는 것(피봇 — 같은 문제 영역 안에서 방향 조정)은 같은 프로젝트에서 영향받는 단계로 되돌아가 이어가고, 전혀 다른 아이디어로 갈아타는 것(폐기 후 새 아이디어)만 새 프로젝트에서 처음부터 시작한다 (§3). 산출물은 `planning/mvp/` 한 폴더 (없으면 생성하고 IdeaValidation부터 시작). 앱 코드는 6단계(MvpBuild)부터 **프로젝트 루트 폴더**(이 저장소의 최상위 폴더 — `planning/` 같은 하위 폴더가 아니라 `package.json`·`src/`가 바로 놓이는 위치)에 생긴다 — 여기서 '앱'은 웹사이트·모바일 앱·데스크톱 프로그램 등 만들려는 소프트웨어 전부를 가리킨다 (이하 문서 전체 동일).

빈 캔버스에서 시작 → 7단계(**IdeaValidation** · **MarketResearch** · ScreenDesign · Prototype · **ProtoRetro** · MvpBuild · **MvpLaunch**) → 통과 기준 4개로 거름:

- **통과 기준① 진입 (IdeaValidation)**: 출처별 mini-interview로 painkiller 신호(항목1·2) 검증 → 빌드 진입 자격 판정. 비용 큰 화면 설계 시작 *전*에 가짜 문제 걸러냄.
- **통과 기준② 경쟁 강도 (MarketResearch)**: Agent deep research로 경쟁자 발견·분석 → 경쟁 강도(없음/약함/강함/매우 강함) 판정 + 차별화 axis·카테고리 재정의 결정. "매우 강함"이면 피봇·폐기 권장, "없음"이면 시장 부재부터 의심(2-market-research §2.4.1).
- **통과 기준③ 프로토 (ProtoRetro)**: 데모 후 인터뷰(N=5+ 권장)로 painkiller 재검증 + Pull 신호 첫 측정 → MvpBuild 진입 go/no-go. 치명 결함 거르기가 목적 — 최종 판정이 아니다.
- **통과 기준④ 최종 (MvpLaunch)**: **광고 집행 전에 동결한 퍼널 기준** vs 실트래픽 실측으로 가설 검증 판정. 이 스킬의 진짜 exit.

왜 최종 판정이 인터뷰가 아니라 실트래픽인가 — 인터뷰는 *말*이라 면전 호의에 물들지만(엄마 테스트 문제), 광고 클릭→가입→사용→결제는 *행동*이라 거짓말을 못 한다. 통과 기준③은 가볍게, 통과 기준④가 무겁게.

**6·7단계(MvpBuild·MvpLaunch)는 의도적으로 얇다** — PRD·Architecture 같은 무거운 단계를 두지 않는다. `screen-design.md`(명세) + prototype zip README(설계도)가 이미 스펙이고, LLM 시대에 이 수준 빌드는 싸다. 문서는 빌더 자신을 위한 최소(계측 정의·런치 기준·판정 기록)만.

**Production은 이 스킬의 범위 밖** — 통과 기준④에서 가설 검증되면 production(매출 스케일업·운영 자동화·다채널 확장)으로 졸업하며, 그건 분기별 KPI·실험 트래킹·운영 도구의 영역이라 이 하네스가 다루지 않는다. 폐기되면 새 아이디어로 새 프로젝트에서 처음부터 새로 시작.

### 1.2 현재 단계 판별

- `planning/mvp/`에서 §2 표 산출물의 존재·완성도를 단계 순으로 검사.
- 비거나 미완성인 가장 빠른 단계 = 진행 대상.
- 사용자가 특정 단계를 콕 집어 지시하면 그 단계로 점프 (평가는 "사용자 명시 점프" 사유로 통과).

### 1.3 단계별 references 라우팅 + 응대 톤

- `references/{N}-{name}.md`를 읽고 그 가이드대로 진행. 각 references는 (a) 사용자 질문 셋, (b) 산출물 스펙, (c) 완료 체크리스트, (d) 좋은/나쁜 예시를 가짐.
- **응대 톤**: 전 단계 공통 — `CLAUDE.md` 대화 스타일 따름. (references가 §1.3으로 이 줄을 참조.)
- **단계명은 내부 코드명** — 유저에게 `ScreenDesign`·`MvpBuild` 같은 코드명을 던지지 않는다. 그 단계 references의 `§1 단계 목표`와 **직전까지 쌓인 산출물 내용**을 바탕으로, 이번 단계에서 뭘 할지를 한국어로 구체적으로 안내한다 (기능·설계처럼 추상적인 단계는 *누가 어떻게 쓰는지*도 곁들임). 고정 문구 X — 매번 그 프로젝트의 실제 맥락에 맞춰 표현한다.
- **IdeaValidation 외 단계는 AI가 먼저 제안** — 안내에 이어 "내가 이전 내용 바탕으로 먼저 정리해볼게, 피드백 주면 수정하고 다시 보여주는 식으로 같이 하자"라고 진행 방식을 밝힌다. **말로 예고만 하지 말고 같은 턴에 실제 초안까지 낸다.** 초안 모드 작업(§1.5)이면 초안을 끝까지 완주해서 낸다. 이후 피드백 → 수정 → 재확인.
  - 예 (MvpBuild 진입): ❌ "MvpBuild 할 차례" / ✅ "프로토타입 반응 봤으니 이제 진짜 서비스로 만들 차례야 — 백엔드·DB 붙이고 배포까지. 스택 몇 가지만 정하면 내가 빌드 완주해서 배포 링크로 보여줄게."
- **옛 단계 evidence는 §2.3 검토 절차 거친 후 사용** — 단계가 쌓이면 새 단계 항목 중 일부가 *신규 작성*이 아니라 *옛 단계 압축·연결*이다. 어떤 옛 내용을 가져올지는 §2.3 4 체크 질문 + 3 라벨(✅·🔵·🟡·footer 🟥)로 결정.

### 1.4 인터뷰 코칭 원칙

각 references의 인터뷰는 **사용자가 처음부터 좋은 답을 만들지 못한다**는 가정으로 작동한다. 기능 명세·가설 같은 산출물은 사람이 처음 써보면 누락·추상·일반론이 섞이는 게 정상이다. AI가 가공·되물음으로 좋은 예 형식까지 끌고 간다.

- **빠진 부분은 배치로 짚어 되묻기.** references의 "좋은 예" 패턴(구체성·검증 가능성·필수 항목 충족)에 사용자 답이 못 미치면 그대로 두지 말고 모자란 항목 체크. **되물음은 주제 단위로 묶어 한 턴에 2~4개** — 하나 묻고 답 받고 또 하나 묻는 직렬 왕복을 피한다. 확인이 잦아지면 사용자가 습관적으로 "응"만 하게 되어 거르는 기능을 잃는다.
- **"잘 모르겠어"는 정상 답.** 사용자가 답을 못하면 AI가 후보 2~3개 제시 → 사용자가 골라서 다듬는 형태로. 빈 칸 채우라고 압박하지 말기.
- **AI가 좋은 예 형식으로 정리해 보여주고 확인 받기.** "이거 맞아? 고칠 데 있어?" — 사용자가 처음부터 형식 갖춰 답할 필요 없음.
- **본질 보존 + 형식만 가공.** 사용자가 답한 의미는 절대 변경 금지. 임의 정보 추가도 금지 — 모르는 부분은 산출물에 `추후 결정: [무엇을 다시 확인할지]`로 명시 후 다음 단계 진입 시 사용자에게 재확인.
- **진행 중 이전 결정 변경 처리.** 단계 진행 중 이전 산출물의 결정이 바뀌어야 함을 감지하면(예: Prototype 만들다 플랫폼이 web → mobile로), AI가 자연어로 영향 범위 짚고 사용자 의향 확인. **영향 작으면**(frontmatter 한 줄 또는 화면 일부) 자연스럽게 부분 업데이트해 진행. **영향 크면**(여러 단계 산출물 영향) 두 옵션 제시: (a) 영향 부분만 손봐서 이어가기, (b) 처음부터 다시 시작하되 직전 산출물을 base로 보존. 전체 단계 재방문 강제 금지. **반영 위치**: 바뀐 아이디어·가설·비목표의 *기술*은 옛 단계 문서 소급 수정이 아니라 *현재(또는 되돌아간) 단계 산출물의 진입 스냅샷*에 한다 (§2.3) — 옛 문서엔 끝에 포인터 한 줄만.

### 1.5 작업 모드 — 인터뷰 모드 vs 초안 모드

각 단계의 작업은 두 모드로 나뉜다. 어느 모드인지 의식하고 진행 방식을 바꾼다.

**인터뷰 모드** — 사용자의 암묵지(문제 인식·타겟 지식·취향·전략)가 소재인 작업. IdeaValidation mini-interview, 플로우 골격 질문, 차별화 결정, 런치 go 선 수치 결정, 통과 기준 go/no-go 판정.
→ 사람과 상호작용이 목적 그 자체다. 배치 질문(§1.4)으로 왕복 횟수만 줄이고, 상호작용 자체를 줄이지 않는다.

**초안 모드** — AI가 산출물을 생산하는 작업. 화면 명세·IA·검증화면 빌드·조사 정리·인터뷰 노트 종합·MVP 빌드·광고 소재·판정 자료 준비.
→ 중간에 잘게 확인받지 않고 **초안을 끝까지 완주한 뒤 완성본으로 한 번에 리뷰**받는다 (draft-then-review). 이유: 중간 승인은 횟수가 쌓이면 습관 승인이 되어 거르는 기능을 잃고, 사용자 시간만 쪼갠다. 리뷰는 완성본이 구체적이라 피드백 품질도 높다.

**초안 모드의 안전판 3개** (완주 ≠ 폭주):

1. **에스컬레이션 기준** — 다음 셋 중 하나면 완주를 멈추고 물어본다:
   - (a) 이전 단계 산출물과 모순 발견 (예: 명세가 비목표와 충돌)
   - (b) 사용자 암묵지 없이는 방향이 갈리는 결정 (타겟·톤·전략 판단)
   - (c) 결과를 되돌리기 비싼 갈림길 (스택 선택·구조 전체가 갈리는 선택)

   그 외의 애매함은 합리적 가정을 박고 `추후 결정: [무엇]`으로 마킹 후 계속.
2. **정지 조건** — 같은 산출물의 자율 재작업은 최대 2회. 그 안에 수렴 안 되면 중간 상태 그대로 들고 사용자에게 (오류 복리 방지).
3. **사람 몫 불변** — 통과 기준 판정(go/no-go·피봇), 실사용자 인터뷰, 화면·데모·배포물 클릭 검증은 초안 모드 대상이 아니다. AI는 그 판단의 *자료*까지만 준비한다.

### 2.1 SoT 매트릭스 — 영역별 진실의 원천

산출물 간 중복·충돌을 피하기 위해 영역별 SoT(Source of Truth) 분리. 각 references는 이 매트릭스를 input/output 경계로 작동.

| 영역                                                                                       | SoT                    | 다른 산출물                                       |
| ------------------------------------------------------------------------------------------ | ---------------------- | ------------------------------------------------- |
| 경쟁자 분석 · 차별화 axis · 카테고리 정의 | **market-research.md** | screen-design는 참조만 |
| **아이디어·가설**(최종 아이디어·타겟·비목표·검증 가설·북극성) | **가장 최신 단계의 진입 스냅샷** (§2.3 — 시작점은 market-research.md §최종 서비스 기획 요약 내용 = 2단계 스냅샷) | 옛 단계 스냅샷은 frozen 기록 — 소급 수정 금지 |

### 2.2 통과 기준③ ProtoRetro — MvpBuild 진입 판정

5번 ProtoRetro에서 **go/no-go 판정**(실서비스 빌드로 갈지): **데모 후 인터뷰 N=5+ 권장**의 행동 의향("베타에 들어가 직접 써볼게") + 페이지 인상 + 운영자 직관 기반. N=5를 못 채워도 판정 가능 — 단 근거가 약함을 proto-retro.md에 명시 (직장+사이드 빌더에게 인터뷰 모집은 현실적으로 어렵고, 최종 검증은 어차피 통과 기준④ 실트래픽이 한다). 결과로 3분기:

- **go → MvpBuild 진입** — 같은 스킬의 6단계로 계속 (`/clear` 후 재호출 권장 — §3.2).
- **no-go → 다른 아이디어** — 만들어 보니 이 아이디어는 가망 없음 → 새 프로젝트에서 처음부터.
- **종료/보류** — 지금 결정 안 함.

"솔루션을 개선해야 한다"는 결론이면 그건 go다 — 솔루션 개선이 곧 MvpBuild·MvpLaunch가 하는 일이라 별도 분기가 아니다. 분기 안내·결정·기록 절차는 `references/5-proto-retro.md` §2.7.

**통과 기준④ (최종)는 MvpLaunch에서** — 실측 퍼널 vs 사전 동결 기준으로 **가설 검증 / 피봇 / 폐기** 판정. 절차는 `references/7-mvp-launch.md` §2.3.

### 2.3 단계 간 SoT 검토 절차

원칙: 단계가 진행되며 산출물이 진화하면 **가장 최신 단계 산출물 = 메인 SoT**. 옛 단계는 *(내용 차원에서)* 최신본의 결론·방향과 포함·상반관계인지 능동 검토 후 evidence로 사용. *명시 인용 여부*는 보조 신호일 뿐 판정 기준 아님. 단계가 쌓이면 새 단계 항목 중 일부가 *신규 작성*이 아니라 *옛 단계 압축·연결* 작업으로 작동.

이 절차는 **MarketResearch(최종 서비스 기획 요약 내용) · ScreenDesign · Prototype · MvpBuild · MvpLaunch 모든 단계 공통**. 각 단계 references는 §2.1 영역별 SoT 매트릭스 + §2.3 단계별 SoT 위계를 input/output 경계로 작동.

#### 진입 스냅샷 — 아이디어·가설은 단계마다 새로 기술 (소급 수정 금지)

단계를 진행할수록 아이디어·가설은 계속 진화한다 — 이게 정상이다. 그 진화를 *옛 단계 문서를 고쳐서* 반영하면 frozen 기록이 living 문서 역할까지 겸하게 되어 SoT가 오염되고, "본문 재정합 TODO" 같은 부채가 쌓인다. 대신 **각 단계 산출물이 자기 시점의 아이디어·가설을 새로 기술**한다:

- **무엇**: 3단계부터 각 단계의 md 산출물(`screen-design.md` · `proto-retro.md` · `mvp-build.md` · `launch-plan.md`) **머리에 `# 진입 스냅샷` 섹션**을 박는다. 내용은 2단계 §최종 서비스 기획 요약 내용과 같은 4개 — 최종 아이디어 한 문단 · 비목표 · 검증 가설 · 북극성(선택).
- **작성 룰**: 단계 진입 시 직전 스냅샷과 대조해 —
  - **변경 없으면** 한 줄: `변경 없음 — [직전 단계 산출물 §섹션] 그대로` (중복 방지).
  - **변경 있으면** 4개 전문 재기술 + `## 변경` 하위에 무엇이 왜 바뀌었는지 한 줄씩.
- **불변 원칙**: 이전 단계 문서는 **frozen 기록** — 방향이 바뀌어도 소급 재정합하지 않는다. 필요하면 옛 문서 *끝에 포인터 한 줄*만 (`→ YYYY-MM-DD 이후 방향은 screen-design.md 진입 스냅샷 참조`).
- **SoT**: 아이디어·가설의 SoT = **가장 최신 단계의 진입 스냅샷** (§2.1 매트릭스). 1·2단계 산출물(idea-validation §최종 확정 아이디어 → market-research §최종 서비스 기획 요약 내용)이 이 체인의 시작이다.
- **진행 중 피봇**: 영향받는 가장 빠른 단계로 되돌아가(§1.4) **그 단계 산출물의 진입 스냅샷을 갱신**한다. 그보다 앞 단계 문서는 안 건드린다. 가설의 형태 변환(문제 가설 → 검증 가설 → 퍼널 가설)은 기존대로 2·5·7단계가 담당 — 스냅샷은 그 *사이 구간*에서 방향 전환이 기술될 자리를 보장하는 장치다.

#### 화면 SoT의 코드 승격 (Prototype 등장 후) — 피봇 시 옛 문서 재작성 금지

Prototype 단계가 zip을 받은 순간부터, §2.1 매트릭스의 화면 영역(플로우·IA·상태·전환 전부)의 SoT는 **`screen-design.md`에서 `prototype-ver-{N}/` 코드로 통째 승격**된다. `wireframes/`가 프로토타입 등장 후 임무를 다하는 것과 같은 논리 — `screen-design.md`도 이 시점부터 *frozen 입력*이지 갱신 대상이 아니다.

**왜**: 프로토타입 코드가 화면에 대해 아는 전부를 담고 있는데, 그 위에 텍스트 명세를 이중으로 유지하면 (1) 둘이 어긋날 때마다 어느 쪽이 진실인지 판단하는 세금이 생기고 (2) 실제로 아무도 다시 안 읽는 문서를 계속 고치는 ceremony가 된다.

**따라서 Prototype 등장 이후 피봇(화면·플로우·IA가 바뀌는 경우)의 기록 방법은 "위 단계로 되돌아가기"가 아니라 다음 2~3가지뿐이다**:

1. **코드로 반영** — 변경을 프로토타입에 직접 반영한다. 규모가 크면 Claude Design 재호출로 새 `prototype-ver-{N+1}/` (기본 경로), 자잘한 예외면 in-place 보정 (4-prototype.md §2.2 그대로).
2. **스냅샷만 새로 기술** — `screen-design.md`를 고치지 않는다. 대신 **아직 안 쓴, 가장 가까운 다음 단계 산출물**(보통 `proto-retro.md` — 아직 파일이 없으면 그 머리에 미리 만들어둠)의 `# 진입 스냅샷`에 바뀐 최종 아이디어·비목표·검증 가설·북극성만 새로 기술한다.
3. **지식 추출 (선택)** — 프로토타입 코드에 하드코딩된 도메인 지식 중 *프로토타입 자체보다 수명이 긴 것*(체크리스트·분기 규칙·가중치·용어 풀이 등, MVP에서 정식 데이터·로직으로 승격될 성격)은 작은 문서로 뽑아 남긴다 (예: `planning/mvp/{주제}-knowledge.md`). 프로토타입은 MvpBuild에서 버려지는 중간물이니, 화면 자체보다 오래 살아야 할 지식만 선별해 코드 밖으로 꺼내는 것 — 화면 명세를 문서로 다시 옮기는 게 아니다.

이 2~3개 산출물(코드 + 스냅샷 + 선택적 지식 추출 문서)이 곧 "피봇이 기록됐다"는 뜻이다. 그 이상 — 옛 문서 본문 소급 재정합, "TODO: 본문 재정합" 같은 부채성 표기 — 은 하지 않는다.

## 3. 산출물 관리

산출물 위치: `planning/mvp/` — vN 없음. 한 폴더에 7단계 산출물 (`idea-validation.md` · `market-research.md` · `screen-design.md` + `wireframes/` · `prototype-ver-{N}/` · `proto-retro.md` · `mvp-build.md` · `launch-plan.md` · `launch-retro.md`).

**앱 코드는 프로젝트 루트 폴더에** — 6단계 MvpBuild부터 `package.json`·`src/` 등이 프로젝트 루트 폴더에 생긴다. `planning/`은 문서 산출물만 (`planning/`은 git 추적 제외 — 로컬 전용. 앱 코드는 git 추적).

**예외 — Prototype 단계는 ver-N 폴더 (zip 패키지)**: Prototype은 사용자가 외부 Claude Design에서 받아 온 **zip 핸드오프 패키지**(압축 해제)가 SoT다 — 진입 reference HTML(데모·정답지, 파일명 가변) + README(설계도) + `src/tokens/`·컴포넌트 소스 (+ `design-system.html`은 있을 수도 없을 수도). 단일 `prototype.md` 없음 — 버전별 폴더로 누적한다. 가장 큰 N의 `prototype-ver-{N}/`가 현재 SoT. 첫 ver는 사용자가 외부 Claude Design 호출 후 zip을 가져옴, 큰 변경은 외부 재호출(새 zip)·자잘한 예외만 reference HTML 직접 보정 (4-prototype.md §2.2). zip 전체는 MvpBuild의 입력. 상세는 `references/4-prototype.md` §2·§5.

**3단계 `wireframes/`는 스캐폴딩**: 손으로 짠 검증 화면은 *플로우가 맞는지 점검*하는 임시 도구라 ver 없이 한 폴더에서 갈아끼운다 — SoT는 `screen-design.md`. Prototype 단계가 Claude Design으로 디자인 완성본을 만들면 검증 화면은 역할을 다한다(폴더는 history로 남겨도 OK).

핵심 원칙: **빈 캔버스에서 시작** — 한 아이디어 = 한 프로젝트. 통과 기준③이 no-go거나 통과 기준④가 폐기면 그 아이디어는 거기서 닫히고, 다른 아이디어는 **새 프로젝트(새 레포)**에서 다시 시작한다 — 같은 레포에서 두 번째 아이디어는 진행하지 않는다. 이유: 서로 다른 아이디어의 산출물이 한 폴더에 섞이면 SoT가 오염되고, 닫은 아이디어의 판정 기록도 깨끗하게 보존해야 하기 때문. **단, 피봇은 새 아이디어가 아니다** — 같은 문제 영역 안에서 방향을 다듬는 것이라 같은 프로젝트에서 영향받는 단계로 되돌아가 이어간다 (2-market-research §2.8 · 7-mvp-launch §2.3). 분기 판정 절차는 §2.2 · `references/5-proto-retro.md` §2.7 · `references/7-mvp-launch.md` §2.3.

### 3.1 영구 의사결정 (ADR)

여러 단계에 걸치는 결정은 **`planning/adr.md` 단일 파일**에 한 섹션씩 누적 — 단계 산출물은 frozen 스냅샷이라 결정의 _이유_·*변경 이력*을 따로 보존. 다음 트리거 감지 시 AI가 한 줄로 확인("이건 ADR감 — 기록할까?"):

- 외부 인프라 선택·교체 (DB·Auth·Storage·Hosting·결제·LLM 등)
- 보안·인증·권한 모델 변경
- 스택 주요 컴포넌트 추가·제거
- 비용 구조에 큰 영향을 주는 결정
- 이전 단계 artifact 결정을 뒤집는 변경

형식: `## ADR-NNNN: <제목> (날짜)` + 상태·변경·이유 3필드. 뒤집을 땐 옛 섹션 보존하고 새 ADR로 supersede. **ADR 아님**: 진행 중 임시 결정(artifact 인라인) · 코드 컨벤션(`.claude/rules/`·`CLAUDE.md`) · 일반 버그·리팩터링.

### 3.2 단계 사이 — `/clear` 후 재호출 권장

매 단계 산출물 저장 직후 (1→2 ~ 6→7 전부), AI는 사용자에게 **`/clear` 후 `/idea-to-mvp` 재호출**을 권장한다. 스킬이 §1.2로 현재 단계 자동 판별하니 잃는 맥락은 없고, references context가 fresh 상태에서 다음 단계 시작 — context 누적·rot 막음. **사용자가 명시 거부하면 그대로 진행.** AI는 자기 context window 크기·사용량을 정확히 모르므로 "필요할 때만" 같은 동적 판단 X — *매 단계 끝에서 결정적으로* 권장.

## 4. 단계 진입 평가와 건너뛰기

각 단계 진입 전 이전 산출물의 **존재**와 **완성도**(해당 references 체크리스트)를 검사. _단계 진입 평가_ = "이전 단계 산출물이 갖춰졌는지 확인하는 잠금장치". 미통과 시 3 옵션:

- (a) 이전 단계 보완 — 안전한 디폴트
- (b) 강행 — 빈 placeholder 산출물 자동 생성 후 진행, 회고에서 재점검
- (c) 산출물 없이 진행 + 회고에서 재점검

선택 사유는 해당 단계 산출물 또는 `proto-retro.md`/`launch-retro.md`에 기록.

**단, 7단계(MvpLaunch) 안 "막1 기준 박기 → 막2 광고 집행" 순서는 건너뛸 수 없다** — launch-plan.md 동결 없이 광고 집행 금지 (7-mvp-launch.md §2.1). 사후 기준은 자기합리화가 되기 때문에 이 잠금은 (b)·(c) 예외 없음.
