## 1. 작동 방식 — 단계 진입과 공통 규칙

### 1.1 스킬 범위 — 아이디어에서 MVP까지, 프로젝트당 1회

이 스킬은 아이디어 하나를 MVP 검증까지 끌고 가는 7단계 과정이다. **한 프로젝트(저장소)는 아이디어 하나만 다룬다** — 피봇(같은 문제 영역 안에서 방향을 다듬는 것)은 같은 프로젝트에서 이어간다. 전혀 다른 아이디어로 갈아타는 경우에는 이 프로젝트를 종료하고, 새 프로젝트(폴더를 새로 생성)를 만들어 처음부터 진행하라고 사용자에게 안내한다. 이 구분과 산출물 위치·관리 규칙은 §3.

어떤 아이디어가 막 생겨난 시점이 이 스킬의 가장 초기 단계다. 거기서 7단계(**IdeaValidation** · **MarketResearch** · ScreenDesign · Prototype · **ProtoRetro** · MvpBuild · **MvpLaunch**)를 거치며 통과 기준 4개로 거른다:

- **통과 기준① 진입 (IdeaValidation)**: 출처별 짧은 인터뷰로 pain killer(진통제) 신호 — 사람들이 이미 임시방편을 만들어 쓰는지, 타겟을 실명으로 꼽을 수 있는지 — 를 확인해 빌드 진입 자격을 판정한다. 비용 큰 화면 설계를 시작하기 *전*에 가짜 문제를 걸러낸다. 판정 절차는 `references/1-idea-validation.md` §2.8·§2.9.
- **통과 기준② 경쟁 강도 (MarketResearch)**: Agent deep research로 경쟁자 발견·분석 → 경쟁 강도(없음/약함/강함/매우 강함) 판정 + 차별화 축(axis)·카테고리 재정의 결정. "매우 강함"이면 피봇·폐기 권장, "없음"이면 시장 부재부터 의심. 판정 절차는 `references/2-market-research.md` §2.4.
- **통과 기준③ 프로토 (ProtoRetro)**: 데모 후 인터뷰(5명 이상 권장)로 pain killer 재검증 + Pull 신호(데모를 본 사람이 자발적으로 "써보고 싶다"고 하는 신호) 첫 측정 → MvpBuild 진입 go/no-go(진행/중단) 판정. 치명 결함 거르기가 목적 — 최종 판정이 아니다. 판정 절차는 `references/5-proto-retro.md` §2.6.
- **통과 기준④ 최종 (MvpLaunch)**: **광고 집행 전에 동결한 퍼널 기준** vs **실제 사용자 행동**의 실측으로 가설 검증 판정. 중요한 데이터는 사용자가 우리가 의도한 행동(가입·사용·결제)을 실제로 했는지이고, 의도하지 않았지만 의미 있는 다른 행동도 함께 본다. 이 스킬이 진짜로 끝나는 지점. 판정 절차는 `references/7-mvp-launch.md` §2.3.

왜 최종 판정이 인터뷰가 아니라 실제 사용자 행동인가 — 인터뷰는 *말*이라, 사람들이 앞에서는 좋게 말해주기 마련이라 실제보다 부풀려진다(맘 테스트(Mom Test)가 지적하는 문제). 반면 광고 클릭→가입→사용→결제는 *행동*이라 거짓말을 못 한다. 통과 기준③은 가볍게, 통과 기준④가 무겁게.

**6·7단계(MvpBuild·MvpLaunch)는 의도적으로 얇다** — PRD·Architecture 같은 무거운 문서 단계를 두지 않는다. `screen-design.md`(명세) + prototype standalone.html(화면·인터랙션 정답지)가 이미 스펙이고, LLM 시대에 이 수준 빌드는 싸다. 문서는 빌더 자신을 위한 최소(계측 정의·런치 기준·판정 기록)만.

**Production은 이 스킬의 범위 밖** — 통과 기준④에서 가설이 검증되면 production(매출 스케일업·운영 자동화·다채널 확장)으로 졸업한다. 그건 분기별 KPI·실험 추적·운영 도구의 영역이라 이 하네스가 다루지 않는다.

### 1.2 현재 단계 판별

- `planning/mvp/`에서 §2.1 표 산출물의 존재·완성도를 단계 순으로 검사.
- 비거나 미완성인 가장 빠른 단계 = 진행 대상.
- 사용자가 특정 단계를 콕 집어 지시하면 그 단계로 점프 (평가는 "사용자 명시 점프" 사유로 통과).
- **Prototype 단계는 `planning/mvp/`가 아니라 프로젝트 루트로 판별**: `planning/mvp/`엔 Prototype 산출물이 없다 — **프로젝트 루트**(이 저장소 최상위 폴더)의 `[projectName]-proto.html`(projectName = 프로젝트 폴더명) 존재 여부 하나로 판별한다. 있으면 "프로토타입 빌드 라운드는 끝났고 지금은 테스트 단계"로 판단해 곧장 사용자 검증 안내 + 5단계(ProtoRetro) 전환 제안으로 이어간다 (`4-prototype.md` §2.3).

### 1.3 단계별 안내 문서(references) 읽기 + 응대 톤

- `references/{N}-{name}.md`를 읽고 그 가이드대로 진행. 모든 references는 같은 골격을 가진다: §1 단계 목표 / §2 진행 절차(§2.1 단계 시작 — 이전 정리 보여주기·이번 할 일 안내·`이전 결론 → 이번 결론` 포함) / §3 완료 체크리스트(+§3.1 다음 단계 핸드오프) / §4 산출물 스펙(+§4.1 템플릿) / §5 좋은·나쁜 예(있는 문서만) / 마지막 섹션은 사용자 응대 톤.
- **응대 톤**: 전 단계 공통 — `CLAUDE.md` 대화 스타일 따름. (references가 §1.3으로 이 줄을 참조.)
- **단계명은 내부 코드명** — 유저에게 `ScreenDesign`·`MvpBuild` 같은 코드명을 던지지 않는다. 그 단계 references의 `§1 단계 목표`와 **직전까지 쌓인 산출물 내용**을 바탕으로, 이번 단계에서 뭘 할지를 한국어로 구체적으로 안내한다 (기능·설계처럼 추상적인 단계는 *누가 어떻게 쓰는지*도 곁들임). 고정 문구를 쓰지 않는다 — 매번 그 프로젝트의 실제 맥락에 맞춰 표현한다.
- **IdeaValidation 외 단계는 AI가 먼저 제안** — 안내에 이어 "내가 이전 내용 바탕으로 먼저 정리해볼게, 피드백 주면 수정하고 다시 보여주는 식으로 같이 하자"라고 진행 방식을 밝힌다. **말로 예고만 하지 말고 같은 턴에 실제 초안까지 낸다.** 초안 모드 작업(§1.5)이면 초안을 끝까지 완주해서 낸다. 이후 피드백 → 수정 → 재확인.
  - 예 (MvpBuild 진입): ❌ "MvpBuild 할 차례" / ✅ "프로토타입 반응 봤으니 이제 진짜 서비스로 만들 차례야 — 백엔드·DB 붙이고 배포까지. 스택 몇 가지만 정하면 내가 빌드 완주해서 배포 링크로 보여줄게."
- **옛 단계 내용은 §2.2 검토 절차를 거친 후 사용** — 어떤 옛 내용을 가져올지는 §2.2의 확인 질문 4개와 표시로 결정한다.

### 1.4 인터뷰 코칭 원칙

각 references의 인터뷰는 **사용자가 처음부터 좋은 답을 만들지 못한다**는 가정으로 작동한다. 기능 명세·가설 같은 산출물은 사람이 처음 써보면 누락·추상·일반론이 섞이는 게 정상이다. AI가 가공·되물음으로 좋은 예 형식까지 끌고 간다.

- **빠진 부분은 배치로 짚어 되묻기.** references의 "좋은 예" 패턴(구체성·검증 가능성·필수 항목 충족)에 사용자 답이 못 미치면 그대로 두지 말고 모자란 항목 체크. **되물음은 주제 단위로 묶어 한 턴에 2~4개** — 하나 묻고 답 받고 또 하나 묻는 직렬 왕복을 피한다. 확인이 잦아지면 사용자가 습관적으로 "응"만 하게 되어 거르는 기능을 잃는다.
- **"잘 모르겠어"는 정상 답.** 사용자가 답을 못하면 AI가 후보 2~3개 제시 → 사용자가 골라서 다듬는 형태로. 빈 칸 채우라고 압박하지 말기.
- **AI가 좋은 예 형식으로 정리해 보여주고 확인 받기.** "이거 맞아? 고칠 데 있어?" — 사용자가 처음부터 형식 갖춰 답할 필요 없음.
- **본질 보존 + 형식만 가공.** 사용자가 답한 의미는 절대 변경 금지. 임의 정보 추가도 금지 — 모르는 부분은 산출물에 `추후 결정: [무엇을 다시 확인할지]`로 명시 후 다음 단계 진입 시 사용자에게 재확인.
- **진행 중 이전 결정 변경 처리.** 단계 진행 중 이전 산출물의 결정이 바뀌어야 함을 감지하면(예: Prototype 만들다 플랫폼이 web → mobile로), AI가 자연어로 영향 범위 짚고 사용자 의향 확인. **영향 작으면**(문서 머리말 한 줄 또는 화면 일부) 자연스럽게 부분 업데이트해 진행. **영향 크면**(여러 단계 산출물 영향) 두 옵션 제시: (a) 영향 부분만 손봐서 이어가기, (b) 처음부터 다시 시작하되 직전 산출물을 base로 보존. 전체 단계 재방문 강제 금지. **반영 위치**: 바뀐 방향은 옛 단계 문서를 고치지 않고 현재(또는 되돌아간) 단계 산출물의 `이전 결론 → 이번 결론` 섹션에 새로 적는다 — 규칙은 §2.2.

### 1.5 작업 모드 — 인터뷰 모드 vs 초안 모드

각 단계의 작업은 두 모드로 나뉜다. 어느 모드인지 의식하고 진행 방식을 바꾼다.

**인터뷰 모드** — 사용자의 암묵지(문제 인식·타겟 지식·취향·전략)가 소재인 작업. IdeaValidation mini-interview, 플로우 골격 질문, 차별화 결정, 런치 go 선 수치 결정, 통과 기준 go/no-go 판정.
→ 사람과 상호작용이 목적 그 자체다. 배치 질문(§1.4)으로 왕복 횟수만 줄이고, 상호작용 자체를 줄이지 않는다.

**초안 모드** — AI가 산출물을 생산하는 작업. 화면 명세·IA 설계·조사 정리·인터뷰 노트 종합·MVP 빌드·광고 소재·판정 자료 준비.
→ 중간에 잘게 확인받지 않고 **초안을 끝까지 완주한 뒤 완성본으로 한 번에 리뷰**받는다. 이유: 중간 승인은 횟수가 쌓이면 습관 승인이 되어 거르는 기능을 잃고, 사용자 시간만 쪼갠다. 리뷰는 완성본이 구체적이라 피드백 품질도 높다.

**초안 모드의 안전판 3개** (완주 ≠ 폭주):

1. **에스컬레이션 기준** — 다음 셋 중 하나면 완주를 멈추고 물어본다:
   - (a) 이전 단계 산출물과 모순 발견 (예: 명세가 비목표와 충돌)
   - (b) 사용자 암묵지 없이는 방향이 갈리는 결정 (타겟·톤·전략 판단)
   - (c) 결과를 되돌리기 비싼 갈림길 (스택 선택·구조 전체가 갈리는 선택)

   그 외의 애매함은 합리적 가정을 박고 `추후 결정: [무엇]`으로 마킹 후 계속.
2. **정지 조건** — 같은 산출물의 자율 재작업은 최대 2회. 그 안에 수렴 안 되면 중간 상태 그대로 들고 사용자에게 (오류 복리 방지).
3. **사람 몫 불변** — 통과 기준 판정(go/no-go·피봇), 실사용자 인터뷰, 화면·데모·배포물 클릭 검증은 초안 모드 대상이 아니다. AI는 그 판단의 *자료*까지만 준비한다.

## 2. 7단계 구성

7단계와 각 단계의 산출물·진실의 원천·안내 문서는 아래 §2.1 표 하나로 본다.

### 2.1 단계 릴레이 표 — 각 단계가 확정하는 것과 진실의 원천(SoT)

7단계는 릴레이다. 각 단계는 이전 단계까지 정리된 내용을 출발점으로 받아 자기 산출물을 만들고, 그 산출물이 다음 단계의 출발점이 된다. 한 번 넘긴 산출물은 더 이상 고치지 않는 기록이고, 뒤 단계는 참조만 한다. 그래서 **어떤 내용의 진실의 원천(SoT — Source of Truth)은 곧 그 내용을 확정한 단계의 산출물**이다. 산출물 위치는 `planning/mvp/` 기준 (4단계만 프로젝트 루트 — §3).

### 2.2 단계 간 SoT 검토 절차

원칙: 단계가 진행되며 산출물이 진화하면 **가장 최신 단계 산출물 = 메인 SoT**. 옛 단계 내용은 최신본의 결론·방향에 포함되는지, 어긋나는지를 능동적으로 검토한 뒤에만 근거로 쓴다. *명시 인용 여부*는 보조 신호일 뿐 판정 기준이 아니다. 단계가 쌓이면 새 단계 항목 중 일부가 *신규 작성*이 아니라 *옛 단계 압축·연결* 작업으로 작동한다.

이 절차는 **MarketResearch(최종 서비스 기획 요약 내용) · ScreenDesign · Prototype · MvpBuild · MvpLaunch 모든 단계 공통**. 각 단계 references는 §2.1 단계 릴레이 표와 이 §2.2 검토 절차를 자기가 읽고 쓸 범위의 경계로 삼는다.

#### 이전 결론 → 이번 결론 — 아이디어·가설은 단계마다 새로 기술 (소급 수정 금지)

**대전제 — 변화는 예외가 아니라 기본이다.** 이 스킬은 애자일하게 굴러간다. 빌더가 아이디어가 속한 도메인을 완벽히 아는 게 아니면(대부분 그렇다), 만들면서 도메인을 알게 되고 **아이디어·비목표·검증 가설·타겟이 매 단계에서 통째로 뒤집힐 수 있다** — screen-design → prototype 사이에서도, prototype 만드는 도중에도, 인터뷰 후에도. 이건 실패가 아니라 이 스킬이 굴러가는 방식 그 자체다. 그러니 "변경 없음"은 여러 케이스 중 하나일 뿐 기본값이 아니다 — 매 단계 "정말 그대로인가?"를 능동적으로 되묻는다.

단계를 진행할수록 아이디어·가설은 이렇게 계속 진화한다. 그 진화를 *옛 단계 문서를 고쳐서* 반영하면, 더 이상 고치지 않기로 한 기록이 계속 바뀌는 문서 역할까지 겸하게 되어 SoT가 오염되고, "본문 재정합 TODO" 같은 부채가 쌓인다. 대신 **각 단계 산출물이 "직전 단계까지의 결론(이전 결론)"과 "이 단계를 지나며 정리된 결론(이번 결론)"을 나란히 새로 기술**한다. 핵심은 *기존엔 이랬는데 이 단계에서 이렇게 정리됐다*가 읽는 순서 그대로 드러나는 것:

- **무엇**: 3단계부터 각 단계의 md 산출물(`screen-design.md` · `proto-retro.md` · `mvp-build.md` · `launch-plan.md`) **머리에 `# 이전 결론 → 이번 결론` 섹션**을 박는다. 다루는 내용은 2단계 §최종 서비스 기획 요약 내용과 같은 4개 — 최종 아이디어 한 문단 · 비목표 · 검증 가설 · 북극성(선택).
- **작성 룰**: 단계 진입 시 직전 단계의 '이번 결론'과 대조해 —
  - **변경 없으면** 한 줄: `변경 없음 — [직전 단계 산출물 §섹션] 그대로` (중복 방지).
  - **변경 있으면** 세 소제목으로 전(前)→후(後)를 나란히 둔다:
    - `## 이전 결론` — 직전 단계까지의 결론 4개(최종 아이디어·비목표·검증 가설·북극성)를 인용·요약. *어느 문서 어느 섹션 기준*인지 출처를 밝힌다. 이전 단계에 없던 항목(예: 검증 가설 섹션 자체가 없었음)은 "없었음"으로 정직하게 적는다 — 억지로 채우지 않는다.
    - `## 이번 결론` — 이 단계에서 정리된 결론 4개를 새로 기술. **이후 이 4개가 아이디어·가설의 SoT.**
    - `## 무엇이 왜 바뀌었나` — 항목별로 `(옛 → 새)` + 사유를 한 줄씩.
- **불변 원칙**: 이전 단계 문서는 **더 이상 고치지 않는 기록**이다 — 방향이 바뀌어도 소급 수정하지 않는다. 필요하면 옛 문서 *끝에 포인터 한 줄*만 (`→ YYYY-MM-DD 이후 방향은 screen-design.md 이전 결론 → 이번 결론 섹션 참조`).
- **SoT**: 아이디어·가설의 SoT = **가장 최신 단계의 '이번 결론'** (§2.1 표 2단계 행). 1·2단계 산출물(idea-validation §최종 확정 아이디어 → market-research §최종 서비스 기획 요약 내용)이 이 체인의 시작이다.
- **진행 중 피봇**: 영향받는 가장 빠른 단계로 되돌아가(§1.4) **그 단계 산출물의 `이전 결론 → 이번 결론` 섹션을 갱신**한다. 그보다 앞 단계 문서는 안 건드린다. 가설의 형태 변환(문제 가설 → 검증 가설 → 퍼널 가설)은 기존대로 2·5·7단계가 담당 — 이 섹션은 그 *사이 구간*에서 방향 전환이 기술될 자리를 보장하는 장치다.

#### 화면 SoT의 코드 승격 (Prototype 등장 후) — 피봇 시 옛 문서 재작성 금지

Prototype 단계가 `[projectName]-proto.html`을 받은 순간부터, 화면 영역(플로우·IA·상태·전환 전부)의 SoT는 **`screen-design.md`에서 이 standalone 파일로 통째 승격**된다 (§2.1 표의 3단계 행 → 4단계 행). `screen-design.md`는 이 시점부터 *더 이상 고치지 않는 입력*이지 갱신 대상이 아니다.

**왜**: 프로토타입 코드가 화면에 대해 아는 전부를 담고 있는데, 그 위에 텍스트 명세를 이중으로 유지하면 (1) 둘이 어긋날 때마다 어느 쪽이 진실인지 판단하는 세금이 생기고 (2) 실제로 아무도 다시 안 읽는 문서를 계속 고치는 ceremony가 된다.

**프로토타입 단계의 '이번 결론' = 최종 HTML 그 자체.** Prototype 단계는 md 문서를 안 만들지만(§3), 결론이 없는 게 아니다 — 그 단계의 '이번 결론'은 **최종 `[projectName]-proto.html`(코드)**이다. 프로토타입을 만들며 아이디어·가설이 통째로 뒤집혔어도, 그 뒤집힌 결과는 **오직 이 HTML에 embody**돼 있다. **그래서 그다음 ProtoRetro의 '이전 결론'은 이 HTML을 읽어서 뽑는다** — 일반 룰("이전 결론 = 직전 단계의 이번 결론")이 여기서도 그대로 성립한다(직전 = 프로토타입, 그 이번 결론 = HTML).

- **무조건 최종 HTML이 SoT** — DEV-NOTES·README·옛 피봇 메모 같은 부차 문서는 결론 도출에 쓰지 않는다. 오직 최종 HTML(그 안의 화면·플로우·카피·기능·데이터 모델)만 근거.
- HTML엔 추상 4개(최종 아이디어·비목표·검증 가설·북극성)가 문장으로 안 적혀 있으니, **AI가 화면·카피·기능에서 4개를 추론 → 사용자에게 보여주고 확인**받는다(§1.4 코칭). 확정된 게 ProtoRetro의 '이전 결론'. HTML이 SoT고, 사용자 확인은 추론의 애매함을 푸는 절차지 외부 문서를 들여오는 게 아니다.

**따라서 Prototype 등장 이후 피봇(화면·플로우·IA가 바뀌는 경우)의 기록 방법은 "위 단계로 되돌아가기"도 "프로토타입 파일을 다시 고치기"도 아니라 다음 2~3가지뿐이다** — Claude Design이 완성해 온 `[projectName]-proto.html`은 받은 그대로 두고 다시 손대지 않는다 (4-prototype.md §2.2):

1. **실제 반영은 MvpBuild에서** — 바뀐 화면·플로우·IA는 go 판정 후 6단계(MvpBuild)에서 실코드로 반영한다. 프로토타입 파일 자체는 사용자 테스트용 중간물이라 반복 수정하지 않는다.
2. **ProtoRetro의 '이전 결론'으로 기록** — `screen-design.md`를 고치지 않는다. 대신 **다음 단계 `proto-retro.md`의 `## 이전 결론`**에, 위 방식대로 *최종 HTML에서 읽어(+사용자 확인)* 바뀐 최종 아이디어·비목표·검증 가설·북극성을 적는다. (프로토타입 도중 피봇은 proto-retro의 '이번 결론'이 아니라 '이전 결론'에 담긴다 — '이번 결론'은 그 프로토타입을 인터뷰한 *뒤* 정리되는 자리이기 때문. 인터뷰 前엔 `추후 결정 — 인터뷰 후`로 비워둔다.)
3. **지식 추출 (선택)** — 프로토타입 코드에 하드코딩된 도메인 지식 중 *프로토타입 자체보다 수명이 긴 것*(체크리스트·분기 규칙·가중치·용어 풀이 등, MVP에서 정식 데이터·로직으로 승격될 성격)은 작은 문서로 뽑아 남긴다 (예: `planning/mvp/{주제}-knowledge.md`). 프로토타입은 MvpBuild에서 버려지는 중간물이니, 화면 자체보다 오래 살아야 할 지식만 선별해 코드 밖으로 꺼내는 것 — 화면 명세를 문서로 다시 옮기는 게 아니다.

이 산출물(`이전 결론 → 이번 결론` 섹션 + 선택적 지식 추출 문서 + MvpBuild의 실코드)이 곧 "피봇이 기록됐다"는 뜻이다. 그 이상 — 옛 문서 본문 소급 재정합, "TODO: 본문 재정합" 같은 부채성 표기 — 은 하지 않는다.

#### 확인 질문 4개 — 옛 문서 내용을 가져와도 되는가

새 단계 산출물을 쓰다 보면 옛 단계 문서의 내용(인터뷰 인용·조사 결과 등)을 근거로 가져오고 싶을 때가 있다. 그런데 옛 문서가 쓰인 뒤 방향이 바뀌었을 수 있다. 그래서 가져오기 전에 아래 4개를 순서대로 확인하고, 결과에 따라 표시를 붙인다.

| 순서 | 확인할 것 | 그렇다면 | 아니라면 |
|---|---|---|---|
| 1 | 그 내용이 **최신 문서에도 그대로 나오나?** (같은 인용·표·사례) | 🔵 가져온다 — 최신 문서의 그 위치를 근거로 적는다 | 2번으로 |
| 2 | 최신 문서가 그 내용을 **버린다고 밝혔나?** | 🟥 가져오지 않는다 — 문서 끝 "버린 내용 목록"에 한 줄 | 3번으로 |
| 3 | 그 내용을 **조금만 넓혀 말해도** 최신 방향과 **어긋나나?** (예: "인터뷰이 A가 그랬다" → "첫 자취생은 그렇다") | 🟥 가져오지 않는다 — 같은 목록에 기록 | 4번으로 |
| 4 | 최신 문서의 **어느 구체 문장·섹션이 그 내용을 품는지 짚을 수 있나?** | 🔵 가져온다 — 넓혀 말한 근거 한 줄 + 그 근거 위치를 함께 적는다 | 🟡 애매함 — 사용자에게 확인 |

**주의**: "버린다고 밝힘"(2번)과 "그냥 언급이 없음"은 다르다. 언급이 없을 뿐이면 3·4번으로 계속 확인한다 — 자동으로 버리지 않는다.

#### 표시 3종 + 버린 내용 목록

산출물 본문에 붙이는 표시:
- **✅ 이번 단계에서 직접** — 지금 단계의 인터뷰·조사에서 직접 얻은 내용
- **🔵 옛 단계에서 가져옴** — 1번 또는 4번 통과. 4번(넓혀 말한 경우)은 근거 한 줄 + 근거 위치 필수 (예: *§4차 deep research §가계약 후 대출 거부 TOP 5 사유 2 "매물 적격성"*). 위치를 못 대면 자동으로 🟡
- **🟡 애매함** — 사용자 확인 전에는 근거로 쓰지 않는다

문서 끝에 두는 목록:
- **🟥 버린 내용 목록** — 2번·3번으로 걸러진 옛 내용을 한 줄씩 기록 (다음 단계·회고에서 재검토)

#### 단계별 적용 — 각 references에서 보강

각 단계 references는 §2.2 절차를 *그대로* 사용. 단계별로 추가 박는 것:
- **단계 항목 × 옛 단계 표준 섹션 종류 매핑** (예: ScreenDesign의 *화면 카피* 항목 → market-research의 *최종 아이디어* 톤 검토)
- 단계별 *표시 판정 적용 예시* 1~2개

매핑·예시는 *섹션 종류(역할)* 로만 명시. 특정 프로젝트 인스턴스의 *구체 섹션 이름·내용*은 references에 박지 않음 — 재사용성 유지.

## 3. 산출물 관리

산출물 위치: `planning/mvp/` — vN 없음. 한 폴더에 6단계 산출물 (`idea-validation.md` · `market-research.md` · `screen-design.md` · `proto-retro.md` · `mvp-build.md` · `launch-plan.md` · `launch-retro.md`). **Prototype 단계만 예외** — 아래 참조.

**앱 코드는 프로젝트 루트 폴더에** — 6단계 MvpBuild부터 `package.json`·`src/` 등이 프로젝트 루트 폴더(이 저장소의 최상위 폴더)에 생긴다. 여기서 '앱'은 웹사이트·모바일 앱·데스크톱 프로그램 등 만들려는 소프트웨어 전부를 가리킨다 (문서 전체 동일). `planning/`은 문서 산출물만 (`planning/`은 git 추적 제외 — 로컬 전용. 앱 코드는 git 추적).

**예외 — Prototype 단계는 `planning/mvp/`에 아무것도 안 둔다**: Prototype 산출물은 **프로젝트 루트**의 `[projectName]-proto.html` 파일 하나뿐이다. 사용자가 외부 Claude Design에서 **standalone.html**(자급자족 단일 파일 — 폰트·이미지·컴포넌트 소스까지 전부 내장)을 받아 와 프로젝트 루트에 이 이름으로 저장하면 그게 SoT. 버전 폴더·zip 압축해제·README 없음 — 받은 그대로 저장하고 다시 손대지 않는다 (4-prototype.md §2.2). 이 파일 전체가 MvpBuild의 입력. 상세는 `references/4-prototype.md` §2·§3.1.

핵심 원칙: **한 아이디어 = 한 프로젝트** — 아이디어가 생겨나면 아무것도 없는 새 프로젝트에서 시작한다. 통과 기준③이 no-go거나 통과 기준④가 폐기면 그 아이디어는 거기서 닫히고, 다른 아이디어는 **새 프로젝트(새 레포)**에서 다시 시작한다 — 같은 레포에서 두 번째 아이디어는 진행하지 않는다. 이유: 서로 다른 아이디어의 산출물이 한 폴더에 섞이면 SoT가 오염되고, 닫은 아이디어의 판정 기록도 깨끗하게 보존해야 하기 때문. **단, 피봇은 새 아이디어가 아니다** — 같은 문제 영역 안에서 방향을 다듬는 것이라 같은 프로젝트에서 영향받는 단계로 되돌아가 이어간다 (2-market-research §2.8 · 7-mvp-launch §2.3). 분기 판정 절차는 `references/5-proto-retro.md` §2.6 · `references/7-mvp-launch.md` §2.3.

### 3.1 영구 의사결정 (ADR)

여러 단계에 걸치는 결정은 **`planning/adr.md` 단일 파일**에 한 섹션씩 누적한다 — 단계 산출물은 더 이상 고치지 않는 기록이라, 결정의 _이유_와 *변경 이력*은 이 파일에 따로 보존한다. 다음 트리거 감지 시 AI가 한 줄로 확인("이건 ADR감 — 기록할까?"):

- 외부 인프라 선택·교체 (DB·Auth·Storage·Hosting·결제·LLM 등)
- 보안·인증·권한 모델 변경
- 스택 주요 컴포넌트 추가·제거
- 비용 구조에 큰 영향을 주는 결정
- 이전 단계 산출물의 결정을 뒤집는 변경

형식: `## ADR-NNNN: <제목> (날짜)` + 상태·변경·이유 3필드. 뒤집을 땐 옛 섹션은 남겨두고 새 ADR가 그것을 대체한다고 적는다. **ADR 아님**: 진행 중 임시 결정(해당 산출물 안에 바로 적음) · 코드 컨벤션(`.claude/rules/`·`CLAUDE.md`) · 일반 버그·리팩터링.

### 3.2 단계 사이 — `/clear` 후 재호출 권장

매 단계 산출물 저장 직후 (1→2 ~ 6→7 전부), AI는 사용자에게 **`/clear` 후 `/idea-to-mvp` 재호출**을 권장한다. 스킬이 §1.2로 현재 단계를 자동 판별하니 잃는 맥락은 없고, 다음 단계를 깨끗한 대화에서 새로 시작하게 된다 — 대화 맥락이 길게 쌓여 판단이 흐려지는 것을 막기 위해서다. **사용자가 명시 거부하면 그대로 진행.** AI는 자기 대화 맥락의 크기·사용량을 정확히 모르므로 "필요할 때만" 같은 판단을 하지 않는다 — *매 단계 끝에서 예외 없이* 권장한다.

### 3.3 어휘 swap·이름 정합 절차

작업 중 핵심 어휘·단계명·산출물명을 바꿔야 할 때(예: 카탈로그→모음, UserFlow→ScreenDesign), 단계 무관 공통 절차:

1. `grep -rn "<옛 단어>" .claude/ planning/`로 흔적 전부 파악
2. 긴 패턴 먼저, 짧은 패턴 나중 순서로 `sed` 일괄 치환
3. 파일·폴더 rename (`mv`)
4. 다른 문서를 *그대로 인용한 대목*은 원문 보존 — 인용 원본을 같이 swap할 때만 인용문도 swap
5. 다시 `grep`으로 잔재 0 확인
6. 산출물 끝 🟥 버린 내용 목록에 기록 (옛 어휘 + 새 어휘 + 사유)

## 4. 단계 진입 평가와 건너뛰기

각 단계 진입 전 이전 산출물의 **존재**와 **완성도**(해당 references 체크리스트)를 검사. _단계 진입 평가_ = "이전 단계 산출물이 갖춰졌는지 확인하는 잠금장치". 미통과 시 3 옵션:

- (a) 이전 단계 보완 — 안전한 디폴트
- (b) 강행 — 빈 placeholder 산출물 자동 생성 후 진행, 회고에서 재점검
- (c) 산출물 없이 진행 + 회고에서 재점검

선택 사유는 해당 단계 산출물 또는 `proto-retro.md`/`launch-retro.md`에 기록.

**단, 7단계(MvpLaunch) 안 "막1 기준 박기 → 막2 광고 집행" 순서는 건너뛸 수 없다** — launch-plan.md 동결 없이 광고 집행 금지 (7-mvp-launch.md §2.1). 사후 기준은 자기합리화가 되기 때문에 이 잠금은 (b)·(c) 예외 없음.
