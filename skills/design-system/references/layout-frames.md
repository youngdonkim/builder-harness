# 프레임·레이아웃 방법론 (frame-first)

## 1. 핵심 개념 — frame · region · content

**이 문서의 용어** (레이아웃·조립 계층). 대부분 은유·관행어라 원어 감각과 다를 수 있어 여기서 못박는다:

| 용어 | 뜻 | 비고 |
|---|---|---|
| frame(프레임) | 셸 + 영역 분할 + 크기 예산 — 콘텐츠와 무관한 뼈대 | 여러 화면이 공유하는 재사용 단위 |
| region(영역) | 프레임 안의 이름 붙은 구역 (위치·크기 예산으로 정의, 콘텐츠 무관) | **개념(자리)이지 특정 DOM 요소가 아님** — 보통 div로 구현되나 CSS Grid 영역·부모 padding으로도 표현 |
| content(콘텐츠) | 영역을 채우는 실제 요소 (리스트·카드·폼…) | |
| section | 콘텐츠 쪽의 의미 묶음 컨테이너(컴포넌트) | region과 층위 다름 — 아래 `region ≠ section` 참조 |
| 크기 예산(size budget) | 영역마다 미리 배정해 둔 크기 한도 | 고정 높이(상단 크롬 52px)든 최대 폭(`--app-max-width` 560px)이든. "예산"=돈이 아니라 "쓸 몫을 미리 정해 둔다"는 은유 |
| gutter | 화면 가장자리와 콘텐츠 사이의 좌우 여백 (예: `--space-screen-x`) | 조판 용어(책 제본 쪽 안쪽 여백 → 칼럼 사이 간격)에서 온 모바일 UI 관행어 |
| 크롬(chrome) | 콘텐츠를 둘러싸고 탐색·조작을 돕는 **틀 UI** (상하단 바·스텝바·닫기 버튼) | 브라우저 이름 아님 — 자동차 크롬 트림 유래(겉을 두르는 마감) |
| canvas | 뷰포트의 배경색 (웹에선 body background) | 프레임이 못 덮은 부분에서 드러남 — 아래 참조 |
| pattern | 프레임+컴포넌트를 화면 유형별로 묶은 검증된 템플릿 | 한 프레임 위에 여러 개 가능 |

**레이아웃 ≠ 화면 디자인.** 레이아웃은 여러 화면이 공유하는 재사용 뼈대이고,
화면은 그 뼈대에 콘텐츠를 채운 인스턴스다. (예: 위저드 10단계 = 화면 10개, 레이아웃 1개)

region ≠ section: **region**은 프레임의 구역 — 위치·크기 예산으로 정의(콘텐츠 무관).
**section**은 콘텐츠 쪽의 의미 묶음 컨테이너(컴포넌트) — region 안에서 콘텐츠를 주제별로 묶으며,
컨테이너 정책(§3)의 적용 대상이다. 위계: frame → region → content(section → 카드/행 → 컴포넌트).

관련 면(surface) 개념 — canvas는 **뷰포트의 배경색**(웹에선 body background)으로,
프레임이 덮지 못한 부분에서 드러난다. 상세는 [naming-taxonomy.md](naming-taxonomy.md)의 표면 위계 참조.

## 2. frame-first 절차 — 콘텐츠부터 쓰지 말 것

1. **프레임 선택**: 프로젝트의 셸 중 하나를 고른다 (프레임 목록 위치는 §6)
2. **영역 예산 확인**: 각 영역의 크기는 layout/frame-budget 토큰으로 정의돼 있다
   (`--app-max-width`, `--nav-height`, gutter, 안전영역 류). 새 영역이 필요하면 토큰부터 추가
3. **컨테이너 정책 결정**: 영역마다 두 가지를 정한다 (§3)
   - 행(row)이냐 카드(card)냐
   - 여백 담당자는 화면(gutter)이냐 카드(마진)냐
4. **그다음에** 콘텐츠(컴포넌트·토큰 선택)로 내려간다

### 2.1 셸 선택 규칙 — 공통 · 변형 · 독립

셸은 세 가지로 관리한다:

| 분류 | 정의 | 구현 |
|---|---|---|
| 공통 셸(base) | 여러 화면 성격이 공유하는 뼈대 | 기본 클래스 (예: `.screen`) |
| 변형 셸(variant) | 뼈대(영역 구성·크기 예산)는 공통 셸과 같고 **표현**(배경·타입 스케일·패딩 톤)만 다름 | 공통 셸 + `--modifier` (예: `.screen--wizard`) |
| 독립 셸(standalone) | **영역 구성 자체가 다름** — 상단 크롬 유무·하단 바 유무·스크롤 모델·히어로형 등 | 별도 클래스 신설 |

판정 기준은 하나: **영역(region) 구성과 크기 예산이 같은가?** 같으면 변형, 다르면 독립.

철칙:
- **공유 셸의 기본 규칙은 공유 자산** — 직접 수정은 그 셸을 쓰는 화면 전부를 동시에
  바꾸겠다는 의도일 때만. 한 성격만 바꿀 땐 반드시 변형(`--modifier`)으로.
  수정 전에 그 셸의 사용처를 검색으로 확인한다.
- 변형 셸이 뼈대까지 갈라지기 시작하면(영역 추가·삭제) **독립 셸로 승격**한다 —
  modifier에 영역 구조 변경을 계속 쌓으면 base가 사실상 두 개가 된다.
- 셸 3종 모두 프레임 인벤토리(§6)에 등록 — 새 화면은 인벤토리 검색부터.
- 변형 셸에서 화면 전체의 토큰을 비례 조정할 땐 자기참조 calc() 패턴을 쓴다
  ([naming-taxonomy.md](naming-taxonomy.md) §2 참조).

## 3. 컨테이너 정책 — 카드 vs 행, 여백 한 겹 원칙

### 3.1 카드 vs 행

**카드는 위젯 컨테이너지, 목록 항목 래퍼가 아니다.**

| 데이터 성격 | 컨테이너 |
|---|---|
| 밀집 데이터·스캔용 목록 | **행(row)** — 리스트/테이블 |
| 자기완결 위젯 (KPI 타일, 차트, 갤러리 항목) | **카드** |

금지: 모든 항목을 카드로 감싸기 · 카드를 페이지 구조 대신 사용 · 카드 중첩.

### 3.2 여백 한 겹 원칙

프레임 안에 카드가 있으면 여백이 **이중으로 쌓인다**:

```
✗ 겹침: 화면 gutter(24) + 카드 자체 여백/패딩(20) = 콘텐츠가 좌우 44px씩 손해
│gutter│ ┌카드──────────┐ │gutter│
│      │ │pad 콘텐츠 pad│ │      │   ← 좁은 화면(앱)에선 치명적
```

규칙: **화면 가장자리와 콘텐츠 사이 여백은 한 겹만.** 영역별로 담당자를 정한다:

- **방식 A — 화면이 gutter 담당** (기본값): 카드·행은 좌우 마진 없이 화면 gutter에 맞춰 풀폭.
  행 목록·텍스트 콘텐츠에 적합.
- **방식 B — 카드 마진이 gutter 역할**: 프레임에 테두리·gutter를 주지 않고,
  카드 자신의 마진이 가장자리 여백을 만든다. 지도·이미지·가로 스크롤 목록처럼
  **엣지-투-엣지(화면 끝까지) 콘텐츠와 카드가 섞이는 영역**에서 가로폭을 최대로 확보.

## 4. 플랫폼별 고려 — 웹 vs 네이티브 앱

React + React Native 등으로 웹·앱을 통합 개발할 때. canvas·frame·gutter의
**원리는 같지만, 보이는 모습과 중요도가 플랫폼마다 다르다.**

### 4.1 canvas: 웹에선 중요, 앱에선 거의 무의미

| | 웹 (데스크톱 브라우저) | 네이티브 앱 |
|---|---|---|
| 뷰포트 vs 프레임 | 뷰포트가 프레임보다 훨씬 넓음 | 프레임 ≈ 뷰포트 (꽉 참) |
| canvas | 프레임 좌우로 **보임** → 색·톤 설계 필요 | 프레임에 가려져 **안 보임** |

→ 규칙: **canvas 토큰은 항상 정의하되, 웹 전용 관심사로 취급한다.**
앱 빌드에서 canvas 관련 결정(색, 프레임 테두리·그림자)이 레이아웃에 영향을 주면 안 된다.

### 4.2 gutter: 플랫폼별로 예산이 달라야 한다

웹은 화면이 넓어 gutter가 여유로워도 되지만, 앱(좁은 뷰포트)에서 gutter가 크면
**콘텐츠 가로폭을 직접 깎아먹는다.** gutter를 토큰으로 뽑아둔 이유가 이것 —
값을 플랫폼별로 다르게 배선할 수 있다 (예: 웹 24px / 앱 16px, 또는 `clamp()`).
카드가 있는 영역이라면 §3.2 한 겹 원칙(방식 B)과 조합해 폭 손실을 줄인다.

## 5. 반응형 계약

프레임 루트에서 브레이크포인트 동작을 **미리 선언**한다
(예: "≤768px에서 사이드 패널은 오버레이로"). 영역별 변화가 임기응변이 아니라 계약이 되게.

## 6. 프로젝트별 프레임·패턴 기록

프레임 인벤토리(셸 종류·영역·예산)와 화면 패턴 목록은 **각 프로젝트 문서**에 기록한다
(위치는 프로젝트 CLAUDE.md 참조). 이 파일은 방법론만 담는다.
