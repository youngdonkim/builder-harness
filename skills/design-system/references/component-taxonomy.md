## 1. 역할 분류 — 6가지

모든 컴포넌트는 6가지 역할 중 하나다.

| 역할 | 하는 일 | 범용 예 |
|---|---|---|
| **action** | 사용자가 누르는 것 | button, FAB, link |
| **input** | 값을 받는 것 | text-field, select, checkbox, slider |
| **container** | 내용을 담는 면 | card, sheet, dialog, section, popover |
| **navigation** | 위치·이동을 다루는 것 | app-nav, tabbar, stepper, breadcrumb |
| **feedback** | 상태를 알리는 것 | toast, progress, skeleton, confetti, badge |
| **display** | 정보를 보여주는 것 | list, table, chip, avatar, thumbnail |

회사마다 부르는 이름이 달라도(toast vs snackbar) 역할은 같다 — 분류는 용어 차이에 오염되지 않는 층.

## 2. 어휘 두 갈래 — 범용 vs 전용

- **범용 컴포넌트**: 어느 제품에나 있는 개념 (button, dialog, sheet, chip, field, stepper…).
  **업계 표준 영어명 강제** — 이름만 보고 누구나(사람·AI) 뭔지 알아야 한다.
- **전용 컴포넌트**: 이 제품에만 있는 고유 개념 — 자유 명명 허용.
  **단, 인벤토리 등록 필수** (이름 + 역할 + 한 줄 설명).

## 3. 동의어 규칙 — 하나의 개념 = 하나의 이름

인벤토리에 개념별 **정본(canonical) 이름**을 지정하고, 동의어를 함께 기록한다:

```
정본: dialog   ← 이렇게도 부름: modal, popup, alert, 모달, 팝업
정본: sheet    ← bottom-sheet, drawer, 바텀시트, 드로어
정본: chip     ← tag, pill, 태그  (badge는 별개 개념으로 예약)
정본: field    ← input, textbox, 입력창
```

요청에 동의어가 등장하면("팝업 띄워줘") 정본으로 번역한 뒤 인벤토리를 찾는다.

## 4. 작성 규약 (BEM·토큰 계약)

- 클래스는 **BEM**: `.블록__요소--변형` (예: `.stepper__close`, `.exit-dialog--warn`)
- 블록 이름 = 인벤토리의 정본 용어 (개념은 표준어, 인스턴스는 용도 접두어 허용: `.exit-dialog`)
- 값은 전부 **semantic 토큰** 참조 — 이 계약을 지킨 컴포넌트만 이식 가능해진다 (§7)

## 5. 결정 트리

```
새 UI가 필요하다
→ ① 인벤토리 검색 (동의어로도)
→ ② 있다 → 재사용. 살짝 다르면 --modifier 변형 추가 (신규 블록 금지)
→ ③ 없다 → 역할 분류(6개 중 하나) → 범용/전용 판정
     → 범용: 업계 표준명 / 전용: 제품 고유명
     → BEM으로 작성, semantic 토큰만 참조
→ ④ 인벤토리에 한 줄 등록
```

## 6. 인벤토리(사전) 형식

- **개념 단위로 등록** — 블록 패밀리(예: `calc-*` 14블록)는 1항목으로 묶는다
- 한 줄 형식: `이름 | 역할 | 범용/전용 | 설명 | 구현(TSX/CSS)`
- **코드가 진실, 사전은 색인** — 컴포넌트 추가 절차의 마지막 단계가 "사전에 한 줄 추가"

## 7. 이식성 — 프로젝트 간 이사

| 대상 | 다음 프로젝트에서 |
|---|---|
| 범용 컴포넌트 | **코드째 복사** — semantic 토큰 계약 덕에 무수정 재사용, foundation 교체 시 자동 리스킨 |
| 전용 컴포넌트 | **새로 조립** (제품 고유 개념이므로) — 단 재료는 범용 부품 + 토큰 |
| 인벤토리 | **새로 작성** (범용 이사분 등록 + 전용 신규 등록) |
